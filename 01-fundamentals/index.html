🧱 Fundamentos de System Design

Los pilares que todo diseño de sistemas respeta. Sin estos conceptos, ningún sistema escala.

Escalabilidad

La capacidad del sistema de manejar más carga aumentando recursos.

⬆️

Vertical Scaling (Scale Up)

Agregar más recursos a un servidor existente: más CPU, más RAM, más disco.


[Server] [BIGGER Server] CPU: 4 cores → CPU: 64 cores RAM: 16 GB RAM: 512 GB ✅ Simple ❌ Límite físico ✅ Sin código nuevo ❌ SPOF ✅ Bajo latencia ❌ Costoso
↔️

Horizontal Scaling (Scale Out)

Agregar más servidores al sistema. Requiere un Load Balancer para distribuir tráfico.


[Load Balancer] / | \ [S1] [S2] [S3] ✅ Sin límite teórico ✅ Alta disponibilidad ❌ Más complejidad ❌ Gestión de estado
💡

Regla práctica: Empieza con vertical scaling (más simple), pero diseña pensando en horizontal desde el día uno. Los sistemas más grandes del mundo (Google, Amazon) escalan horizontalmente.

Métricas Clave

Los números que definen si tu sistema funciona bien.

⏱️ Latencia

Tiempo en completar una operación. P50, P95, P99 (percentiles). Una API buena tiene P99 < 200ms.

📈 Throughput

Operaciones por unidad de tiempo (QPS, RPS). Cuánto puede manejar el sistema en paralelo.

🟢 Disponibilidad

% de tiempo que el sistema está operativo. 99.9% = 8.7hs/año caído. 99.99% = 52min/año.

🏋️ Durabilidad

Los datos persisten incluso ante fallas. "Once written, never lost." Critical para storage.

🔒 Consistencia

Todos los nodos ven los mismos datos al mismo tiempo. Strong vs eventual consistency.

🔧 Tolerancia a Fallos

El sistema continúa funcionando aunque fallen algunos componentes. Redundancia.

CAP Theorem

El principio más importante en sistemas distribuidos. Formulado por Eric Brewer en 2000.

⚠️

La regla: En un sistema distribuido con partición de red (Partition Tolerance), solo puedes elegir entre Consistency o Availability. No ambas al mismo tiempo.

CAP THEOREM ___________ / \ Consistency Availability (C) (A) Todos los nodos El sistema responde ven los mismos siempre, aunque la datos al mismo data no esté 100% tiempo actualizada Partition Tolerance (P) El sistema sigue funcionando aunque falle la red entre nodos ┌─────────────────────────────────────────────┐ │ CP (Consistency + Partition Tolerance) │ │ → MongoDB, Redis, HBase, ZooKeeper │ │ → Sacrifica: disponibilidad │ │ │ │ AP (Availability + Partition Tolerance) │ │ → Cassandra, DynamoDB, CouchDB │ │ → Sacrifica: consistencia fuerte │ │ │ │ CA (Consistency + Availability) │ │ → Solo en sistemas NO distribuidos │ │ → Bases de datos tradicionales (MySQL) │ └─────────────────────────────────────────────┘
🏦

Cuándo elegir CP

Sistemas financieros, reservas de vuelos, inventario. Cuando los datos incorrectos causan problemas graves. Prefieres estar offline que mostrar data desactualizada.

📱

Cuándo elegir AP

Redes sociales, DNS, shopping carts. Cuando es aceptable mostrar data levemente desactualizada. La disponibilidad importa más que la perfección.

Modelos de Consistencia

Del más estricto al más relajado.

🔒 Strong Consistency

Después de un write, todos los reads ven el nuevo valor. La forma más estricta. Alta latencia. Usado en: Google Spanner, ZooKeeper.

📖 Read-your-writes Consistency

Después de un write, el mismo cliente siempre lee su propio write. Otros clientes pueden ver datos viejos por un tiempo.

🔄 Eventual Consistency

El sistema eventualmente llegará a consistencia. Los writes se propagan a todos los nodos con un delay. Usado en: DynamoDB, Cassandra, DNS.

Strong Consistency: Write → [Node A] → sync → [Node B] → sync → [Node C] ✅ Todos ven el dato nuevo inmediatamente ❌ Alta latencia (hay que esperar sync) Eventual Consistency: Write → [Node A] [Node B: viejo] [Node C: viejo] ↓ (después de un tiempo) [Node B: nuevo] [Node C: nuevo] ✅ Baja latencia (responde inmediatamente) ❌ Ventana de inconsistencia

Estimaciones de Escala (Back-of-the-envelope)

Cómo calcular rápidamente los requerimientos de un sistema en una entrevista.

📐 Ejemplo: Twitter — Calcular QPS y Storage

300M usuarios activos mensuales · 50% usan diariamente · Promedio 2 tweets/día · 50% de tweets tienen media (1MB promedio)

USUARIOS ACTIVOS DIARIOS (DAU): 300M × 50% = 150M DAU WRITE QPS (tweets por segundo): 150M × 2 tweets / 86,400 segundos ≈ 3,500 QPS Peak QPS = 3,500 × 2 = ~7,000 QPS STORAGE PARA TEXTO: 150M × 2 × 280 bytes = 84 GB/día = ~30 TB/año STORAGE PARA MEDIA: 150M × 2 × 50% × 1 MB = 150 TB/día ← ¡Enorme! READ/WRITE RATIO: Twitter: 100:1 (más reads que writes) READ QPS = 7,000 × 100 = 700,000 QPS CONCLUSIÓN: → Necesitas agresivo caching para reads → CDN para media → Multiple DB shards para writes
Otros Conceptos Fundamentales

🔀 Load Balancing

Distribuir tráfico entre múltiples servidores. Algoritmos: Round Robin, Weighted Round Robin, Least Connections, IP Hash. Puede ser L4 (TCP/UDP) o L7 (HTTP).

🌐 CDN (Content Delivery Network)

Red de servidores distribuidos geográficamente que cachean contenido estático (imágenes, JS, CSS, videos) cerca del usuario final. Reduce latencia dramáticamente.

🧩 Microservicios

Dividir un sistema en servicios pequeños, independientes, con dominio propio. Escalan de forma independiente. Mayor complejidad operacional.

📮 Async Processing

En vez de procesar síncronamente, encolar la tarea y procesarla en background. Mejora throughput y user experience. Trade-off: eventual consistency.

🔁 Redundancia

Tener múltiples instancias de cada componente para eliminar SPOF (Single Point of Failure). Active-Active o Active-Passive.

💾 Data Partitioning

Horizontal (sharding): dividir filas en múltiples tablas. Vertical: dividir columnas. Directory-based: lookup service para saber dónde está cada dato.

CAP en Práctica — Framework de Decisión

Basado en HelloInterview · Cómo elegir entre Consistency y Availability en sistemas reales.

💡

La clave: En cualquier sistema distribuido, la Partition Tolerance es inevitable — las redes fallan. La decisión real es: cuando hay una partición, ¿prefiero rechazar requests (CP) o servir datos potencialmente desactualizados (AP)?

🎫

Cuándo elegir Consistency (CP)

Cuando los datos incorrectos generan problemas reales de negocio:

  • Ticket booking: dos usuarios no pueden comprar el mismo asiento
  • Inventario e-commerce: no vender stock que no existe
  • Trading financiero: precios stale generan pérdidas reales
  • Transferencias bancarias: el saldo debe ser exacto
PostgreSQL · Redis · ZooKeeper
📱

Cuándo elegir Availability (AP)

Cuando estar online importa más que la perfección de los datos:

  • Redes sociales: ver un perfil con 5 minutos de delay es aceptable
  • Streaming: descripción de un video levemente desactualizada, ok
  • Reviews/ratings: brief staleness tolerable
  • DNS: propagación lenta es aceptable
Cassandra · DynamoDB · CouchDB

🔀 Estrategia Híbrida (Lo que usan los mejores sistemas)

Muchos sistemas modernos aplican Consistency para operaciones críticas y Availability para features secundarios. Ejemplo en una app de citas: matching de usuarios → CP (no puedes matchear a alguien dos veces). Visualización del perfil → AP (mostramos el perfil aunque esté levemente desactualizado).

EJEMPLO: Sistema de Reservas de Vuelos CRÍTICO (CP — Consistency): Reserva de asiento → Transacción ACID en PostgreSQL Si hay partición → Rechaza el request (mejor error que double-booking) ┌────────────────────────────────────┐ │ User A y B intentan asiento 14A │ │ Transacción con lock │ │ Solo uno gana → el otro recibe │ │ HTTP 409 Conflict │ └────────────────────────────────────┘ NO CRÍTICO (AP — Availability): Historial de vuelos del usuario → Cassandra Si hay partición → Mostramos lo que tenemos (eventual consistency) ┌────────────────────────────────────┐ │ Tu historial puede estar │ │ desactualizado por segundos/min │ │ → No es un problema de negocio │ └────────────────────────────────────┘
Consistent Hashing

Basado en "System Design Interview" Alex Xu Cap. 5 · El algoritmo que resuelve el resharding.

⚠️

El problema del hash simple: Si tienes 4 servidores y usas hash(key) % 4, agregar un 5to servidor invalida el 80% de los keys. Consistent hashing minimiza ese impacto a ~1/N keys.

CONSISTENT HASHING — El Ring: Imagina un anillo de 0 a 2^32: 0 / 2^32 │ S3 ← ───┼─── → S0 / │ \ / Ring (hash \ / space) \ S2 ←─────────────────── S1 Cada KEY se mapea al PRIMER servidor en sentido horario. Key "user:123" → hash → posición en el ring → Server S1 CUANDO AGREGAS UN SERVIDOR S4: Solo los keys entre S3 y S4 se mueven → ~1/N impacto (25% no 80%!) VIRTUAL NODES (para distribución uniforme): Cada servidor físico tiene múltiples posiciones en el ring: S0: posiciones 0, 45, 90, 180 S1: posiciones 30, 75, 120, 210 → Distribución más pareja

✅ Ventajas

Agregar/quitar servidores solo redistribuye ~K/N keys (K = total keys, N = servidores). Sin downtime para resharding. Usado en Amazon DynamoDB, Apache Cassandra, Discord.

🎯 Casos de Uso

Distributed caches (Redis Cluster). Load balancing cuando necesitas afinidad. Distributed databases para determinar en qué shard va cada key.
