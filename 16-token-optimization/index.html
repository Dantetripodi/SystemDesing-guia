🪙 Claude Code: Token Optimization

10 repos de GitHub que reducen el consumo de tokens en Claude Code hasta un 90%. Cómo instalarlos, configurarlos y combinarlos para maximizar tu productividad y minimizar costos.

💰 Hasta -90% tokens 🔧 10 repos curados ⚡ Setup inmediato 🧠 Context Engineering 🚀 Claude Code Pro
0 · ¿Por qué importan los tokens en Claude Code?
El contexto es limitado y caro — cada token que ahorras es dinero y calidad
💡

Context Window de Claude: Claude tiene una ventana de contexto de 200K tokens (claude-sonnet-4). Cuando usás Claude Code en proyectos grandes, el solo hecho de leer archivos puede consumir decenas de miles de tokens por sesión. Con el plan Pro (~$20/mes) tenés límites de uso. Optimizar tokens = más trabajo en menos tiempo + menos interrupciones por rate limits.

📊 El problema de escala

Un repo de 50K líneas de código tiene ~500K tokens solo en archivos fuente. Cargar todo el contexto por cada pregunta es ineficiente. Claude necesita contexto relevante, no todo el codebase.

💸 Costo real

Claude Sonnet 4: $3 / 1M tokens input. Una sesión de trabajo de 2 horas sin optimización puede consumir 500K–1M tokens. Con las herramientas correctas: 50K–100K tokens. Ahorro: 80–90%.

🧠 Calidad del output

Menos tokens no es solo costo — es calidad. Con contexto más limpio y relevante, Claude entiende mejor tu intent y comete menos errores. Context = signal, no noise.

🔑 Las 4 palancas de optimización

  • Indexación semántica: En vez de leer archivos enteros, buscar solo los fragmentos relevantes (semantic search sobre el codebase)
  • Compresión de contexto: Comprimir conversaciones largas antes de que roten, preservando lo importante
  • CLAUDE.md optimizado: Instructions compactas, no verbosas — cada línea que ponés en CLAUDE.md se carga en cada turno
  • Tool call efficiency: Agrupar operaciones, evitar reads redundantes, cachear resultados de herramientas

📦 1 · Context Mode — El más poderoso (-60 a -90% tokens)

Plugin MCP para Claude Code que indexa tu codebase en una base de datos FTS5 (Full Text Search) en SQLite. En vez de leer archivos completos, Claude busca solo los fragmentos relevantes. Es el que mayor impacto tiene de todos.

⚡

Setup Context Mode

# Buscar en GitHub: "context-mode claude code"
# o el plugin que ya tenés activo en esta sesión

# 1. Instalar el plugin MCP
npm install -g context-mode-mcp
# o via Claude Code settings

# 2. Configurar en .claude/settings.json
{
  "mcpServers": {
    "context-mode": {
      "command": "context-mode-mcp",
      "args": ["--index-dir", ".context-mode"]
    }
  }
}

# 3. Indexar el proyecto
# Claude Code lo hace automáticamente
# o forzar re-index con /ctx-index
🔍

Cómo funciona internamente

# Sin Context Mode:
Read(file) → 2000 tokens por archivo
× 20 archivos = 40,000 tokens / query

# Con Context Mode:
ctx_search("implementación de auth")
→ devuelve 5 fragmentos relevantes
→ ~500 tokens / query

# Flujo interno:
1. Indexa archivos → SQLite FTS5
2. Claude llama ctx_search(query)
3. FTS5 retorna top-N fragmentos
4. Claude ve solo lo relevante
5. 90% menos tokens en reads
✅

Tip de uso: Después de instalar Context Mode, cambiá tu flujo de trabajo. En vez de decirle a Claude "lee el archivo X", decí "buscá cómo está implementado X en el proyecto". Context Mode va a encontrar los fragmentos relevantes sin cargar el archivo completo.

📦 2 · claude-context (Zilliz) — RAG sobre tu codebase (-70% tokens)

Solución de Zilliz (creadores de Milvus) que implementa RAG (Retrieval-Augmented Generation) sobre tu codebase usando Milvus Lite como vector database embebida. Chunks el código, genera embeddings, y hace búsqueda semántica en cada consulta.

🚀

Setup Zilliz claude-context

# GitHub: github.com/zilliztech/claude-context

pip install claude-context milvus-lite

# Indexar proyecto
claude-context index ./src \
  --chunk-size 200 \
  --overlap 50 \
  --model text-embedding-3-small

# Iniciar servidor MCP
claude-context serve --port 3000

# Agregar a Claude Code
# .claude/settings.json:
{
  "mcpServers": {
    "codebase": {
      "url": "http://localhost:3000"
    }
  }
}
🧠

Diferencia vs Context Mode

# Context Mode: FTS5 (keyword-based)
ctx_search("getUserById")
→ busca texto exacto
→ rápido, sin GPU

# Zilliz: Semántico (embedding-based)
vector_search("get user from database")
→ entiende intención
→ encuentra aunque no tenga
   las palabras exactas
→ necesita embeddings (API call)

# Cuándo usar cada uno:
- FTS5: repos con naming claro
- Semántico: repos legacy / docs
- Ideal: ambos en paralelo

📦 3 · RTK (Retrieval Toolkit) — Retrieval eficiente para Claude Code

Toolkit que provee múltiples estrategias de retrieval: BM25, dense retrieval, y hybrid search. Diseñado específicamente para el flujo de Claude Code, con caché de resultados y re-ranking automático.

📐

Setup RTK

# Buscar: "RTK retrieval toolkit claude"
npm install @rtk/claude-retrieval

# Configurar en CLAUDE.md:
## Retrieval Strategy
- Use RTK for all file lookups
- Prefer semantic search over Read
- Cache results within session

# En .rtk/config.json:
{
  "strategy": "hybrid",
  "bm25_weight": 0.4,
  "dense_weight": 0.6,
  "top_k": 5,
  "rerank": true,
  "cache_ttl": 3600
}
⚖️

Hybrid Search en práctica

# BM25 (keyword):
# + Exacto con nombres de funciones
# + Sin dependencias externas
# - No entiende sinónimos

# Dense (embedding):
# + Semántico, entiende intención
# + Funciona con código comentado
# - Requiere embedding API

# Hybrid (RTK default):
score = 0.4*bm25 + 0.6*dense
# Mejor de ambos mundos
# Re-ranker cross-encoder
# al final mejora aún más

# Resultado típico:
# Query: "validación de formularios"
# → 5 fragmentos relevantes
# → 300 tokens vs 15K del archivo

📦 4 · Token Savior — Compresión automática de contexto (-40% tokens)

Hook de Claude Code que intercepta conversaciones largas y las comprime automáticamente antes de que el contexto rote. Preserva los puntos clave y descarta el ruido acumulado en sesiones largas.

💾

Setup Token Savior

# GitHub: buscar "token-savior claude"
pip install token-savior

# Configurar hook en Claude Code
# .claude/settings.json:
{
  "hooks": {
    "prePrompt": "token-savior compress",
    "thresholdTokens": 50000,
    "preserveRatio": 0.3
  }
}

# O manual vía comando:
token-savior --session current \
  --compress \
  --keep-decisions \
  --keep-code-changes
🔄

Qué preserva al comprimir

# Token Savior prioriza:
✅ Decisiones arquitectónicas
✅ Código escrito/modificado
✅ Errores encontrados y fixes
✅ Requirements del usuario
✅ Estado actual del task

# Token Savior descarta:
❌ Exploraciones fallidas
❌ Salidas de herramientas verbosas
❌ Repetición de contexto
❌ Pensamiento en voz alta
❌ Confirmaciones intermedias

# Resultado típico:
# 80K tokens → 25K tokens
# preservando todo lo útil

📦 5 · Caveman Claude — Prompts primitivos = menos tokens (-30%)

Filosofía radical: eliminar toda la verbosidad de los prompts e instrucciones. Instrucciones directas como "caveman": sin palabrería, sin explicaciones innecesarias. Sorprendentemente efectivo.

🗿

El estilo Caveman

# ❌ Prompt verboso (180 tokens):
"Por favor, podrías ayudarme a revisar
el archivo de configuración para ver si
hay algún problema con las variables de
entorno? Me gustaría entender si están
bien configuradas para producción."

# ✅ Prompt Caveman (25 tokens):
"check config/env — prod ready?"

# ❌ CLAUDE.md verboso (500 tokens):
"You are an expert software engineer
with 20 years of experience. Always
think step by step..."

# ✅ CLAUDE.md Caveman (80 tokens):
"Stack: Node 20, Postgres 16, React 18
Tests: vitest. Lint: biome.
No comments. No summaries."
📝

CLAUDE.md Caveman template

# Ejemplo CLAUDE.md ultra-compacto:
# Project
[nombre] — [descripción en 1 línea]
Stack: [tech stack]

# Rules
- [regla 1 en <5 palabras]
- [regla 2 en <5 palabras]
- No explain obvious code
- No trailing summaries
- Edit, don't rewrite

# Commands
test: [comando]
lint: [comando]
dev: [comando]

# Resultado: CLAUDE.md de 300 tokens
# vs CLAUDE.md típico de 2000+ tokens
# Se carga EN CADA TURNO → ahorro masivo
⚠️

Importante: Caveman Claude funciona mejor con modelos capaces como Claude Sonnet/Opus. Con modelos pequeños, la brevedad puede generar ambigüedad. La clave es ser conciso y preciso, no simplemente corto.

📦 6 · claude-token-efficient — Formato estructurado para Claude (-35%)

Biblioteca que reformatea automáticamente tus prompts al formato más eficiente para Claude. Elimina redundancia, usa XML tags optimizadas, y estructura el contexto según las mejores prácticas de Anthropic.

🏗️

Setup claude-token-efficient

pip install claude-token-efficient

# Uso básico en Python:
from claude_efficient import optimize

prompt = optimize("""
  [tu prompt aquí]
""", target="claude-sonnet")

# CLI para analizar prompts:
claude-efficient analyze prompt.txt
# Output:
# Original: 850 tokens
# Optimized: 520 tokens (-39%)
# Suggestions:
#  - Remove 3 redundant phrases
#  - Consolidate 2 repetitive rules
#  - Use XML tags for structure
🏷️

XML Tags que Claude entiende mejor

# ✅ Formato eficiente para Claude:
<task>Refactor auth module</task>

<context>
File: src/auth/index.ts (120 lines)
Issue: No refresh token logic
</context>

<constraints>
- Keep existing interface
- Add tests
</constraints>

# Claude procesa XML tags de forma
# más eficiente que prosa libre.
# La biblioteca convierte tu texto
# a este formato automáticamente.

📦 7 · token-optimizer-mcp — Servidor MCP que optimiza tool calls (-40%)

Servidor MCP que actúa como proxy entre Claude Code y tus herramientas. Comprime outputs de herramientas, agrupa calls relacionadas, y cachea resultados para evitar reads redundantes.

🔌

Setup token-optimizer-mcp

# GitHub: buscar "token-optimizer-mcp"
npm install -g token-optimizer-mcp

# .claude/settings.json:
{
  "mcpServers": {
    "optimizer": {
      "command": "token-optimizer-mcp",
      "args": [
        "--compress-threshold", "500",
        "--cache-ttl", "1800",
        "--truncate-logs", "100"
      ]
    }
  }
}

# Envuelve todas las tool calls
# automáticamente al activarlo
📉

Qué optimiza en tool calls

# Sin optimizer:
Bash("npm test")
→ 3000 tokens de output completo
   incluyendo warnings, coverage,
   y stack traces de tests pasados

# Con optimizer:
Bash("npm test")
→ 180 tokens (summary)
  "15 passed, 0 failed (2.3s)"
  + errores relevantes si los hay

# También cachea:
Read("package.json") → cache hit
# No re-lee si no cambió

# Trunca logs largos:
# Solo primeros N + últimos N lines

📦 8 · claude-token-optimizer — Middleware API para proyectos (-45%)

Middleware que se sitúa entre tu código y la API de Anthropic. Implementa prompt caching, request deduplication, y compresión de mensajes. Ideal si construís aplicaciones sobre la API de Claude.

🔀

Setup como middleware API

pip install claude-token-optimizer

# En tu código Python:
from claude_optimizer import OptimizedClient
import anthropic

client = OptimizedClient(
    anthropic.Anthropic(),
    cache_system_prompt=True,
    deduplicate_tools=True,
    compress_history=True,
    max_history_tokens=10000
)

# Úsalo igual que el cliente normal:
response = client.messages.create(
    model="claude-sonnet-4-6",
    messages=[{"role": "user",
               "content": "..."}]
)
# Caching y compresión son automáticos
💾

Prompt Caching integrado

# claude-token-optimizer agrega
# cache_control automáticamente:

{
  "role": "user",
  "content": [
    {
      "type": "text",
      "text": "Large system context...",
      "cache_control": {
        "type": "ephemeral"
      }
    }
  ]
}

# Resultado en factura Anthropic:
# Input tokens: 10,000
# Cache write: 10,000 (primera vez)
# Cache read: 10,000 × 10 calls
# Costo cache read: 10% del normal
# → Ahorro: 90% en llamadas siguientes

📦 9 · code-review-graph — Reviews eficientes con grafos de dependencias (-30%)

Construye un grafo de dependencias de tu codebase y lo usa para dar a Claude solo los archivos directamente relacionados con el cambio que estás revisando. Elimina contexto irrelevante en code reviews y debugging.

🕸️

Setup code-review-graph

# GitHub: buscar "code-review-graph"
pip install code-review-graph

# Analizar dependencias del proyecto:
crg analyze ./src --lang typescript

# Antes de un review, generar contexto:
crg context --file src/auth/login.ts \
            --depth 2 \
            --output context.md

# Claude Code lee context.md en vez
# de explorar el repo manualmente

# O como MCP tool:
crg serve  # expone get_context(file)
🗺️

Cómo reduce el contexto

# Sin code-review-graph:
# Claude lee el repo entero para
# entender qué afecta login.ts
# → 50+ archivos → 100K tokens

# Con code-review-graph:
# login.ts importa: auth-service.ts
#                   user-model.ts
#                   jwt-util.ts
# Es importado por: app.ts
# → Solo esos 5 archivos → 8K tokens

# El grafo se construye 1 vez
# y se actualiza con --watch

📦 10 · token-optimizer (CLI) — Análisis y compresión de prompts (-25%)

CLI de propósito general para analizar, medir y comprimir prompts. Útil para entender exactamente cuántos tokens consume cada parte de tu workflow y encontrar dónde optimizar.

🔬

Setup token-optimizer CLI

npm install -g @anthropic/token-optimizer
# o buscar: "token-optimizer anthropic"

# Contar tokens de un archivo:
token-count CLAUDE.md
# → 847 tokens

# Analizar una sesión completa:
token-optimizer analyze \
  --session ~/.claude/sessions/latest \
  --breakdown

# Comprimir un prompt:
token-optimizer compress prompt.txt \
  --target 60%  \
  --preserve-code \
  --output compressed.txt

Gestión de sesiones largas

- Claude empieza a olvidar decisiones

- "Como mencioné antes..." repetido
- Rate limit warnings

1. /compact  → comprime la sesión
2. Anotá el estado actual en un .md
3. Nueva sesión con ese .md como contexto

            implementá el módulo users,
            que usa auth para verificar
