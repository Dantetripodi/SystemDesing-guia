🔐 Seguridad en System Design

El módulo más ignorado en preparación y el que más diferencia a un senior. Autenticación, autorización, cifrado, ataques y defensa en sistemas distribuidos.

Authentication vs Authorization

El primer concepto que todo interviewer espera que distingas correctamente.

🪪

Authentication (AuthN)

¿Quién sos? — Verificar la identidad del usuario o servicio.

  • Username + password
  • OAuth 2.0 / OpenID Connect
  • Biometría, hardware tokens
  • Certificados mTLS (entre servicios)
  • API Keys
🛡️

Authorization (AuthZ)

¿Qué podés hacer? — Determinar qué acciones le están permitidas.

  • RBAC — Role-Based Access Control
  • ABAC — Attribute-Based Access Control
  • ACL — Access Control Lists
  • Policy-based (Open Policy Agent)
  • Scopes en OAuth
⚠️

Error clásico en entrevistas: Confundir autenticación con autorización. "El usuario está logueado" (AuthN) no significa "puede editar este recurso" (AuthZ). Siempre verificar ambos en el backend — nunca confiar solo en el frontend.

Sessions vs JWT — El Trade-off fundamental

La decisión de arquitectura más común en AuthN. Cada approach tiene sus casos ideales.

SESSION-BASED (Stateful): 1. Usuario hace login 2. Server crea una sesión en memoria/Redis → sessionId = "abc123" 3. Server devuelve cookie: Set-Cookie: sessionId=abc123 4. Cada request → Server consulta: ¿existe abc123? ¿quién es? ┌────────┐ login ┌────────┐ store ┌───────┐ │ Client │──────────→│ Server │─────────→│ Redis │ │ │←── cookie ─│ │ │ DB │ └────────┘ └────────┘ └───────┘ Cada request: cookie → Redis lookup → User data ✅ Revocación instantánea (borra la sesión del server) ✅ Menos datos viajan en cada request ❌ Stateful → sticky sessions o Redis compartido ❌ No funciona bien en microservicios ───────────────────────────────────────────────────── JWT-BASED (Stateless): 1. Usuario hace login 2. Server genera JWT firmado con secret/private key 3. Cliente guarda el JWT (localStorage o cookie httpOnly) 4. Cada request → envía JWT en Authorization header 5. Server valida la firma — NO necesita DB lookup ┌────────┐ login ┌────────┐ │ Client │───────────→│ Server │ (verifica con secret key) │ │←─── JWT ───│ │ └────────┘ └────────┘ Cada request: JWT → validar firma → payload → User data ✅ Stateless → horizontal scaling sin Redis compartido ✅ Funciona perfectamente en microservicios ✅ Claims embebidos (userId, roles, scopes) ❌ NO se puede revocar antes de que expire (el problema) ❌ Payload viaja en cada request (aunque está firmado)

Estructura de un JWT

header.payload.signature — Cada parte es Base64URL encoded.

Revocación de JWT

Solución: access token de vida corta (15min) + refresh token de vida larga (7d) en blacklist/whitelist en Redis.

Dónde guardar el JWT

Cookie httpOnly + Secure + SameSite=Strict es más seguro que localStorage (vulnerable a XSS).

// JWT — Anatomía completa

// HEADER (algoritmo + tipo)
{
  "alg": "RS256",   // RSA con SHA-256 (asimétrico — mejor que HS256)
  "typ": "JWT"
}

// PAYLOAD (claims)
{
  "sub": "user:12345",    // subject — userId
  "iss": "auth.myapp.com", // issuer
  "aud": "api.myapp.com",  // audience
  "exp": 1716239022,       // expiry timestamp (15 min)
  "iat": 1716238122,       // issued at
  "roles": ["user", "admin"],
  "scope": "read:profile write:posts"
}

// SIGNATURE — el server verifica que el token no fue modificado
// RS256: firmado con private key, verificado con public key
// HS256: mismo secret para firmar y verificar (peor para microservicios)
OAuth 2.0 + OpenID Connect (OIDC)

El estándar de la industria para delegación de acceso. Pregunta frecuente en Meta, Google y Stripe.

💡

Conceptos clave: OAuth 2.0 es un framework de autorización (delegar acceso). OIDC (OpenID Connect) agrega autenticación encima de OAuth — el "Login con Google" usa OIDC.

Actores en OAuth 2.0

Resource Owner: el usuario que tiene los datos.
Client: la aplicación que quiere acceso (tu app).
Authorization Server: el que emite tokens (Google, Auth0, tu propio server).
Resource Server: la API que protege los datos (Google Drive API).

Redirect a Google con: │ │ client_id, redirect_uri, │ │ scope, state, code_challenge (PKCE) │ ↓ │ ┌──────────────────────┐ │ │ Google Auth Server │ │ │ (Authorization Srv) │ │ └──────────────────────┘ │ │ 3. User consiente │ │ 4. Google → redirect con ?code=AUTH_CODE │ ↓ │ ┌────────┐ │ │ User │──── 4. code en URL ────────────────────→│ │Browser │ │ └────────┘ ↓ 5. Tu App intercambia: code + client_secret → Google Server 6. Recibe: access_token refresh_token id_token (OIDC) 7. Tu App usa access_token para llamar Google Drive API ✅ code viaja por el browser (URL) ✅ tokens van server-to-server (más seguro) ✅ PKCE previene code interception attacks

Client Credentials Flow

Machine-to-machine. No hay usuario. Tu microservicio A obtiene token para llamar microservicio B. El client_id + client_secret se intercambian directamente por un access_token. Usado en backend APIs, cronjobs, pipelines.

Scopes — Principio de menor privilegio

El access_token tiene scopes que limitan qué puede hacer. Ejemplo: scope="read:email" no le da acceso a calendar. Diseña tus APIs con scopes granulares desde el día uno.

RBAC vs ABAC — Modelos de Autorización

Cómo decidir qué puede hacer cada usuario en sistemas complejos.

RBAC — Role-Based Access Control: Usuario → tiene Roles → Roles tienen Permisos user: alice roles: [admin, editor] admin role: [read:*, write:*, delete:*] editor role: [read:posts, write:posts] ¿Puede alice borrar un post? → alice tiene "admin" → admin tiene "delete:*" → ✅ SÍ Casos de uso: Sistemas internos, SaaS con plans (free/pro/enterprise) Pros: Simple de entender y auditar Cons: Explosión de roles en sistemas complejos ("role explosion") ───────────────────────────────────────────────────── ABAC — Attribute-Based Access Control: Decisión basada en atributos del user, resource, y contexto. Política: "ALLOW if user.department == resource.department AND time < 18:00" user.attributes: { dept: "engineering", clearance: "L3", location: "AR" } resource.attributes: { dept: "engineering", sensitivity: "internal" } context: { time: "14:00", ip: "10.0.0.1" } Evaluación: ¿user.dept == resource.dept? → ✅ SÍ → ALLOW Casos de uso: Fintech, healthcare, sistemas con compliance complejo Pros: Extremadamente flexible y granular Cons: Complejo de implementar y debuggear

Cifrado — En Tránsito y En Reposo

Los dos vectores de ataque que todo sistema debe cubrir.

🔒

Encryption at Rest

Datos cifrados cuando están almacenados (discos, backups, snapshots).

  • AES-256 — estándar de la industria para datos en reposo
  • Full disk encryption — AWS EBS, Azure Disk
  • Field-level encryption — cifrar columnas específicas (PII)
  • Key Management — AWS KMS, HashiCorp Vault, GCP Cloud KMS
  • Rotación de keys sin downtime
🔐

Encryption in Transit

Datos cifrados mientras viajan por la red. Nadie puede leerlos si interceptan el tráfico.

  • TLS 1.3 — obligatorio en cualquier sistema moderno
  • HTTPS — TLS aplicado a HTTP
  • mTLS — mutual TLS, ambos lados se autentican (microservicios)
  • Certificate pinning (mobile apps)
  • HSTS header — fuerza siempre HTTPS
TLS HANDSHAKE (simplificado) — TLS 1.3: Client Server │ │ │──── ClientHello ──────────────────→│ │ (TLS version, cipher suites, │ │ client random) │ │ │ │←─── ServerHello ──────────────────│ │ (chosen cipher suite, │ │ server random, │ │ certificate con public key) │ │ │ │ Client verifica el certificado │ │ contra una CA (Certificate Auth) │ │ │ │──── Finished (con session key) ───→│ │ (key derivada de los randoms │ │ usando el algoritmo acordado)│ │ │ │ ══════ CANAL CIFRADO ══════════ │ │←════════ Datos cifrados ══════════│ TLS 1.3 mejoras sobre 1.2: ✅ 1-RTT handshake (antes 2-RTT) ✅ 0-RTT para reconexión (session resumption) ✅ Forward Secrecy obligatorio (ECDHE)

Symmetric vs Asymmetric Encryption

Simétrico (AES): misma key para cifrar y descifrar. Muy rápido. Problema: cómo intercambiar la key de forma segura.
Asimétrico (RSA, ECDSA): par de keys — public key cifra, private key descifra. Lento, pero resuelve el intercambio de keys.
En la práctica: TLS usa asimétrico para acordar una key simétrica de sesión → luego usa simétrico para el resto (mejor performance).

Top Ataques & Defensa — OWASP Top 10

Los vectores de ataque más comunes y cómo los sistemas senior los mitigan.

SQL Injection

Ataque: Input malicioso modifica la query SQL. '; DROP TABLE users; --
Defensa: Parameterized queries / Prepared statements siempre. NUNCA concatenar strings para SQL. ORM ayuda pero no es suficiente solo.

XSS — Cross-Site Scripting

Ataque: Inyectar JS malicioso en páginas vistas por otros usuarios. Roba cookies, sesiones, datos.
Defensa: Content Security Policy (CSP) header. Escapar output HTML. httpOnly cookies (JS no puede leerlas). Sanitizar todo input de usuarios.

CSRF — Cross-Site Request Forgery

Ataque: Sitio malicioso hace requests a tu API usando las cookies del usuario logueado.
Defensa: CSRF tokens en forms. SameSite=Strict o SameSite=Lax en cookies. Verificar Origin/Referer header. Para APIs REST con JWT en header → CSRF no aplica.

DDoS — Distributed Denial of Service

Ataque: Inundar el sistema con tráfico para hacerlo inaccesible.
Defensa en capas: CDN absorbe tráfico de ataque (Cloudflare, AWS Shield). Rate limiting por IP. WAF (Web Application Firewall). Auto-scaling. CAPTCHA para humanos. Anycast routing para distribuir el ataque.

MITM — Man in the Middle

Ataque: Interceptar comunicación entre cliente y servidor para leer o modificar datos.
Defensa: TLS 1.3 obligatorio. HSTS (HTTP Strict Transport Security). Certificate Transparency logs. Certificate pinning en apps móviles críticas.

Broken Authentication

Ataques: Credential stuffing, brute force, session hijacking.
Defensa: Rate limiting en login. MFA (Multi-Factor Auth). bcrypt/Argon2 para hashear passwords (nunca MD5/SHA1). Tokens de sesión con entropía suficiente. Logout seguro (invalida sesión en server).

Secrets Management

Cómo manejar credenciales, API keys y secretos en sistemas de producción.

⚠️

Regla #1: Nunca hardcodear credenciales en el código fuente. Nunca commitear .env files con secrets. En 2024, el 14% de los repos públicos en GitHub tienen secrets expuestos.

Environment Variables

Mínimo viable. Secretos en variables de entorno, nunca en el código. Suficiente para proyectos chicos. Problema: difícil de rotar y auditar a escala.

AWS Secrets Manager / GCP Secret Manager

Almacenamiento centralizado, rotación automática, integración nativa con IAM. La opción estándar en clouds. Versioning automático.

HashiCorp Vault

Solución open source multi-cloud. Dynamic secrets (genera credenciales de DB por cada servicio, con TTL). Ideal para infraestructura compleja o on-premise.

DYNAMIC SECRETS con HashiCorp Vault: Sin Vault: App → tiene "admin:password123" hardcodeado → todos los servicios usan la misma credencial → un leak compromete todo Con Vault Dynamic Secrets: ServiceA → solicita credencial a Vault → Vault genera: usuario: "svc-a-abc123" / password: "xyz789" / TTL: 1 hora ServiceB → solicita credencial a Vault → Vault genera: usuario: "svc-b-def456" / password: "uvw012" / TTL: 1 hora ✅ Credenciales únicas por servicio ✅ Expiran automáticamente (TTL) ✅ Rotación sin downtime ✅ Auditoría completa de quién pidió qué
Zero Trust Architecture

"Never trust, always verify" — El modelo de seguridad para arquitecturas modernas de microservicios.

El viejo modelo (Perimeter Security)

Si estás adentro de la red corporativa, confiamos en vos. Problema: un atacante que entra a la red puede moverse libremente. El 75% de los ataques vienen de adentro o de credenciales comprometidas.

Zero Trust — Principios Core

1. Verify explicitly — Autenticar y autorizar cada request, sin importar origen.
2. Least privilege — Acceso mínimo necesario. No más.
3. Assume breach — Diseñar como si el atacante ya estuviera adentro. Limitar blast radius.

ZERO TRUST en Microservicios — Service Mesh con mTLS: Tradicional (trust interno): ServiceA ──────────── HTTP ──────────────→ ServiceB (sin autenticación entre servicios) Zero Trust con mTLS (mutual TLS): ServiceA ServiceB ┌─────────┐ presenta certificado ┌─────────┐ │ App │←──────────────────────→│ App │ │ + │ verifica certificado │ + │ │ Envoy │ de ambos lados │ Envoy │ │ Sidecar │ Conexión cifrada │ Sidecar │ └─────────┘ E2E └─────────┘ │ │ └───────────────────────────────────┘ Istio / Linkerd (Service Mesh) gestiona certificados automáticamente ✅ Cada servicio debe autenticarse ✅ Todo el tráfico interno está cifrado ✅ Políticas de red granulares (ServiceA puede llamar ServiceB pero NO ServiceC)
Hashing de Passwords

Los passwords nunca se guardan en texto plano. Nunca.

Lo que NUNCA hay que hacer

Guardar en texto plano. Usar MD5 o SHA1 (crackeables con rainbow tables en segundos). Usar SHA256 sin salt (vulnerable a rainbow tables).

// ✅ bcrypt — estándar mínimo aceptable
import bcrypt

# Al guardar el password:
salt = bcrypt.gensalt(rounds=12)  # work factor — aumenta costo computacional
hashed = bcrypt.hashpw(password.encode(), salt)
# guardar 'hashed' en la DB

# Al verificar:
bcrypt.checkpw(attempt.encode(), hashed_from_db)  # True/False

// ✅✅ Argon2id — el más recomendado hoy (ganó Password Hashing Competition 2015)
# Resiste ataques GPU y side-channel
# Parámetros: memory cost, time cost, parallelism
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=2)
hash = ph.hash(password)
ph.verify(hash, attempt)

Rate Limiting como Herramienta de Seguridad

No es solo performance — es la primera línea de defensa.

Token Bucket

Un bucket tiene capacidad N. Se llena a velocidad R tokens/segundo. Cada request consume 1 token. Si no hay tokens → 429 Too Many Requests. Permite bursts controlados.

Sliding Window Counter

Ventana de tiempo que se desliza. Más preciso que fixed window (evita el double-hit en el boundary). Implementación con Redis ZADD/ZREMRANGEBYSCORE.

Estrategias de Rate Limiting por Layer

Por IP: Defensa DDoS básica (1000 req/min por IP).
Por Usuario/API Key: Business logic (100 req/min en plan gratuito).
Por Endpoint: Más estricto en endpoints sensibles (login: 5 intentos/min).
Global: Límite total del sistema para proteger backends.
En el API Gateway: Kong, AWS API Gateway, Nginx — rate limiting centralizado.

Preguntas de Entrevista — Seguridad

Las preguntas que aparecen en entrevistas semi-senior y senior en FAANG.

"Diseñá un sistema de autenticación para una app con millones de usuarios"

Respuesta esperada: OAuth 2.0 + OIDC para terceros. JWT con access token (15min) + refresh token (7d, httpOnly cookie). Redis para blacklist de refresh tokens revocados. bcrypt/Argon2 para passwords. Rate limiting en /login (5 req/min por IP). MFA opcional vía TOTP (Google Authenticator). Audit logs de todos los logins.

"¿Cómo asegurás comunicación entre microservicios?"

mTLS entre servicios (Istio/Linkerd gestiona certificados automáticamente). Service accounts con least privilege. API Gateway valida tokens externos. Secrets en HashiCorp Vault o AWS Secrets Manager. Network policies en Kubernetes (solo servicios autorizados pueden comunicarse). Audit logging de todas las llamadas entre servicios.

"¿Cómo previenen que un empleado malicioso acceda a datos de usuarios?"

Data classification (PII, sensitive, public). Field-level encryption de datos sensibles — ni siquiera DBAs pueden leer los datos en texto plano. Break-glass procedures para acceso de emergencia (con aprobación y audit). Audit logs inmutables (CloudTrail). RBAC estricto — developers de producción no deben tener acceso directo a datos de usuarios. Data masking en logs (nunca loggear passwords, tokens, o PII completos).
