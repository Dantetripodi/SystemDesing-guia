🏛️ Patrones de Arquitectura

Los patrones que los ingenieros senior usan para diseñar sistemas escalables y mantenibles.

Monolito vs Microservicios

La decisión arquitectónica más importante y más discutida en la industria.

🗿

Monolito

Todo el código en una sola unidad deployable. Un solo proceso, una sola base de datos.


┌─────────────────────┐ │ MONOLITO │ │ ┌───────────────┐ │ │ │ Auth Module │ │ │ │ Users Module │ │ │ │ Orders Mod. │ │ │ │ Payment Mod. │ │ │ └───────────────┘ │ │ [Single DB] │ └─────────────────────┘

✅ Simple de desarrollar y debuggear
✅ Sin latencia de red entre módulos
✅ Transacciones simples

❌ Escala completa aunque solo un módulo necesita escalar
❌ Deploy lento (todo o nada)
❌ Un bug puede tirar todo

🧩

Microservicios

Cada dominio de negocio es un servicio independiente con su propia base de datos.


[Auth Svc] [User Svc] [DB] [DB] \ / [API Gateway] / \ [Order Svc] [Pay Svc] [DB] [DB]

✅ Escala independiente por servicio
✅ Deploy independiente
✅ Teams autónomos por servicio

❌ Complejidad operacional alta
❌ Latencia de red entre servicios
❌ Transacciones distribuidas

⚠️

Martin Fowler dice: "Don't start with microservices. Start monolithic, then break into services when you have scaling or team problems." Amazon, Netflix, Uber empezaron como monolitos.

Event-Driven Architecture

Los servicios se comunican mediante eventos en lugar de llamadas directas.

SÍNCRONO (Acoplado): Order Service ──HTTP→ Payment Service ──HTTP→ Inventory Service (espera respuesta de cada uno, todos deben estar UP) EVENT-DRIVEN (Desacoplado): Order Service → Event: "OrderPlaced" → Message Bus │ ┌───────────────┼───────────────┐ ↓ ↓ ↓ Payment Svc Inventory Svc Email Svc (procesa (reserva (envía pago) stock) email)

📢 Event Notification

Publicar que algo ocurrió. El evento lleva poca data. Los consumidores consultan el source si necesitan más info. Muy desacoplado.

📦 Event-Carried State Transfer

El evento lleva toda la data necesaria. El consumidor no necesita consultar al emisor. Más data en el evento, menos requests.

📜 Event Sourcing

El estado del sistema ES la secuencia de eventos. No guardas el estado actual, guardas todos los eventos. Puedes reconstruir cualquier estado pasado.

CQRS — Command Query Responsibility Segregation

Separar el modelo de escritura (Commands) del modelo de lectura (Queries).

SIN CQRS: Un solo modelo para todo ┌─────────────────────────────────┐ │ Order Model (Read + Write) │ │ → Difícil de optimizar ambos │ └─────────────────────────────────┘ CON CQRS: Modelos separados ┌──────────────────┐ ┌──────────────────────────┐ │ COMMAND SIDE │ │ QUERY SIDE │ │ (Write) │ │ (Read) │ │ │ │ │ │ PlaceOrder() │─→──│ GetOrderDetails() │ │ CancelOrder() │ │ GetUserOrders() │ │ UpdateOrder() │ │ SearchOrders() │ │ │ │ │ │ [Write DB] │ │ [Read DB - Optimizada] │ │ Normalized │ │ Denormalized / Cache │ └──────────────────┘ └──────────────────────────┘ │ ↑ └──── Event/Sync ───────────┘

✅ Cuándo usar CQRS

Cuando reads y writes tienen requerimientos muy distintos. Sistemas con muchos más reads que writes. Cuando necesitas múltiples "vistas" del mismo dato.

❌ Cuándo NO usar CQRS

CRUD simples. Sistemas pequeños. Cuando la complejidad añadida no se justifica. CQRS aumenta la complejidad significativamente.

Saga Pattern

Gestionar transacciones distribuidas en microservicios sin necesidad de 2PC.

💡

El problema: En microservicios, cada servicio tiene su propia DB. No puedes hacer un atomic transaction entre Order DB, Payment DB e Inventory DB. Saga resuelve esto con compensaciones.

SAGA COREOGRAPHY (Eventos): 1. Order Svc → Event: OrderCreated 2. Payment Svc → escucha → cobra pago → Event: PaymentCompleted 3. Inventory Svc → escucha → reserva stock → Event: StockReserved 4. Order Svc → escucha → marca pedido como confirmado Si el pago FALLA: Payment Svc → Event: PaymentFailed Order Svc → escucha → cancela orden (compensación) SAGA ORCHESTRATION (Orquestador central): ┌─────────────────────┐ │ Order Orchestrator │ └──────────┬──────────┘ 1. → Payment Svc: "Cobra pago" 2. ← PaymentSvc: OK 3. → Inventory Svc: "Reserva stock" 4. ← Inventory Svc: FAIL 5. → Payment Svc: "Reversa pago" (compensación)
Circuit Breaker

Patrón para prevenir fallos en cascada cuando un servicio dependiente está caído.

STATES del Circuit Breaker: CLOSED (normal) → OPEN (falla detectada) → HALF-OPEN (probando) ↑ │ └────────────── Si funciona, vuelve a CLOSED ──┘ ┌─────────────────────────────────────────────────────┐ │ CLOSED: Pasa todas las requests │ │ Si tasa de error > umbral → pasa a OPEN │ │ │ │ OPEN: Rechaza todas las requests inmediatamente │ │ (fail fast, sin esperar timeout) │ │ Después de X segundos → pasa a HALF-OPEN │ │ │ │ HALF-OPEN: Deja pasar algunas requests para probar │ │ Si funcionan → CLOSED │ Si fallan → OPEN │ └─────────────────────────────────────────────────────┘
# Circuit Breaker con tenacity (Python)
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=4, max=10)
)
def call_payment_service(order_id):
    response = requests.post("/payments", json={"order_id": order_id})
    response.raise_for_status()
    return response.json()
BFF — Backend for Frontend

Un backend especializado por tipo de cliente (mobile, web, TV).

SIN BFF: Mobile App ──────┐ Web App ─────────┼──→ [Generic API] → Services Smart TV ────────┘ (mismo API para todos, difícil de optimizar) CON BFF: Mobile App ──→ [Mobile BFF] ─────┐ Web App ────→ [Web BFF] ──────────┼──→ [Services] Smart TV ───→ [TV BFF] ───────────┘ (cada BFF agrega, filtra y adapta la data para su cliente)

¿Por qué usar BFF?

Mobile necesita menos data (menos bandwidth). Web puede pedir más. TV tiene UX diferente. Cada BFF puede estar optimizado para su cliente específico. Netflix, SoundCloud y Spotify usan este patrón.

Outbox Pattern

Resolver el dual-write problem entre tu base de datos y un message broker. Patrón crítico para senior.

⚠️

El problema: Necesitas guardar en DB Y publicar un evento en Kafka atómicamente. Si guardas en DB y Kafka se cae antes del publish, el evento se pierde. Si publicas primero y la DB falla, tienes un evento sin el dato. ¿Cómo garantizas que ambas cosas pasan o ninguna?

DUAL-WRITE PROBLEM (lo que NO hacer): Order Service: 1. INSERT INTO orders (id, status) VALUES (123, 'created') 2. kafka.publish("order.created", {order_id: 123}) ← puede fallar aquí Si kafka falla después del INSERT → inconsistencia irreparable ───────────────────────────────────────────────────────── OUTBOX PATTERN (la solución): MISMA transacción de DB: BEGIN; INSERT INTO orders (id, status) VALUES (123, 'created'); INSERT INTO outbox (event_type, payload, created_at) VALUES ('order.created', '{"order_id": 123}', NOW()); COMMIT; ← atómico: ambos o ninguno Outbox Relay Service (proceso separado): LOOP: rows = SELECT * FROM outbox WHERE processed = false ORDER BY created_at FOR row IN rows: kafka.publish(row.event_type, row.payload) UPDATE outbox SET processed = true WHERE id = row.id ┌──────────────┐ ┌──────────────┐ ┌─────────────────┐ │ Order Svc │ │ Outbox DB │ │ Outbox Relay │ │ │ │ (misma DB │ │ (polls outbox │ │ INSERT order│───→│ del svc) │───→│ y publica en │ │ INSERT event│ │ │ │ Kafka) │ └──────────────┘ └──────────────┘ └────────┬────────┘ │ ↓ ┌────────────┐ │ Kafka │ └────────────┘ ✅ Atomicidad garantizada (transacción DB) ✅ At-least-once delivery (el relay reintenta si Kafka falla) ✅ Orden garantizado por created_at ⚠️ At-least-once: los consumidores deben ser idempotentes

Implementaciones del Outbox Relay

Polling: El relay consulta la tabla outbox cada N segundos. Simple pero agrega carga a la DB.
Debezium (CDC — Change Data Capture): Lee el WAL (Write-Ahead Log) de PostgreSQL/MySQL directamente. Más eficiente, sin polling. Detecta cambios a nivel de replication log. La solución más usada en producción a escala.

Strangler Fig Pattern

Migrar incrementalmente de un monolito a microservicios sin downtime. Cómo lo hicieron Amazon y Netflix.

El nombre

El higo estrangulador es una planta que crece alrededor de un árbol huésped. Con el tiempo, el árbol original muere y el higo lo reemplaza. El patrón es igual: los microservicios crecen alrededor del monolito hasta reemplazarlo.

STRANGLER FIG — Migración paso a paso: FASE 1 — Estado inicial: Client → [Monolito] (todo en un solo sistema) FASE 2 — Agregar un Facade/Proxy delante: Client → [API Gateway / Proxy] → [Monolito] (sin cambios aún) FASE 3 — Extraer el primer servicio: Client → [API Gateway] → /users → [User Microservice] (nuevo) → /orders → [Monolito] → /products → [Monolito] FASE 4 — Extraer más servicios gradualmente: Client → [API Gateway] → /users → [User Microservice] → /orders → [Order Microservice] → /products → [Monolito] (reducido) FASE 5 — El monolito desaparece: Client → [API Gateway] → /users → [User Svc] → /orders → [Order Svc] → /products → [Product Svc] → [Monolito eliminado] Cada paso es deployable independientemente. Si algo falla, el Proxy puede hacer rollback redirigiendo al monolito.

✅ Por qué funciona

No hay un "big bang rewrite" (que históricamente falla). La migración es incremental. Cada servicio extraído puede validarse en producción antes de extraer el siguiente. El negocio sigue funcionando durante todo el proceso.

Casos reales

Amazon tardó años en migrar su monolito. Netflix también fue incremental. Airbnb usó este patrón para separar el monolito Ruby on Rails. La clave: el API Gateway/Proxy es el punto de control que permite la transición gradual.

Sidecar Pattern

Un proceso auxiliar que vive junto al servicio principal para agregar funcionalidad cross-cutting.

SIDECAR en Kubernetes: ┌─────────────────────────────────────────┐ │ Pod │ │ ┌─────────────────┐ ┌──────────────┐ │ │ │ Tu App │ │ Sidecar │ │ │ │ (main container│ │ (Envoy Proxy│ │ │ │ port 8080) │ │ port 15001)│ │ │ │ │ │ │ │ │ │ No sabe nada │ │ mTLS │ │ │ │ de mTLS, │ │ Tracing │ │ │ │ tracing, ni │ │ Metrics │ │ │ │ load balancing │ │ Retries │ │ │ └─────────────────┘ └──────────────┘ │ │ Comparten red y storage │ └─────────────────────────────────────────┘ Todo el tráfico pasa por el Sidecar. Tu app no necesita código extra para estas features. Ejemplos de Sidecars: · Envoy Proxy → usado por Istio/Linkerd para service mesh · Fluentd → recolección de logs sin tocar la app · Vault Agent → inyección de secrets sin código de Vault en la app
