SQL vs NoSQL

La decisión más frecuente en entrevistas de system design.

CaracterísticaSQL (Relacional)NoSQL
SchemaRígido, predefinidoFlexible, dinámico
ConsistenciaACID (fuerte)BASE (eventual, mayormente)
EscalabilidadVertical principalmenteHorizontal nativamente
Queries complejas✅ JOINs, agregaciones❌ Limitado
Velocidad de escrituraModeradaMuy alta
Casos de usoFinanzas, e-commerce, ERPSocial, IoT, real-time, ML
EjemplosPostgreSQL, MySQL, AuroraMongoDB, Cassandra, DynamoDB
🎯

En una entrevista: Por default usa SQL. Justifica NoSQL solo cuando tengas una razón clara: escala masiva de escrituras, schema que cambia mucho, o datos no relacionales (grafos, timeseries, documentos).

Tipos de Bases de Datos NoSQL
📄

Document (MongoDB, Firestore)

Datos como documentos JSON. Flexible schema. Ideal para catálogos de productos, perfiles de usuario, CMS. Queries ricas dentro del documento.

Flexible schema
🔑

Key-Value (Redis, DynamoDB)

La más simple y rápida. Key → Value. Ideal para caching, sesiones, leaderboards, contadores. Redis es in-memory con persistencia opcional.

Ultra rápida
📊

Wide-Column (Cassandra, HBase)

Columnas por fila, no schema fijo. Diseñada para escrituras masivas y distribución global. Instagram, Netflix usan Cassandra. Excelente para timeseries.

Escala masiva
🕸️

Graph (Neo4j, Neptune)

Nodos y relaciones como estructura principal. Ideal para redes sociales, detección de fraude, sistemas de recomendación, knowledge graphs.

Relaciones
📈

Time Series (InfluxDB, TimescaleDB)

Optimizada para datos con timestamp. IoT, métricas de infraestructura, analytics financiero. Compresión automática de datos históricos.

Métricas · IoT
🔍

Search (Elasticsearch, Typesense)

Indexación invertida para full-text search. No es un DB principal, sino un complemento. Sincronizas datos desde tu DB principal.

Full-text search
ACID vs BASE
🔒

ACID (SQL)

Atomicity: La transacción es todo o nada.
Consistency: La DB pasa de un estado válido a otro.
Isolation: Las transacciones no se interfieren.
Durability: Una vez committed, persiste aunque falle el servidor.

Garantía fuerte
🌊

BASE (NoSQL)

Basically Available: El sistema siempre responde.
Soft State: El estado puede cambiar con el tiempo.
Eventually Consistent: Los datos eventualmente son consistentes.

Alta disponibilidad
Database Sharding

Partir la base de datos horizontalmente para escalar más allá de un solo servidor.

SIN SHARDING (un solo DB): ┌──────────────────────┐ │ Users DB │ ← Límite físico (~100M rows) │ user_id 1..100M │ └──────────────────────┘ CON SHARDING (múltiples DBs): user_id % 3 == 0 → Shard 0 user_id % 3 == 1 → Shard 1 user_id % 3 == 2 → Shard 2 ┌────────────────┐ ┌────────────────┐ ┌────────────────┐ │ Shard 0 │ │ Shard 1 │ │ Shard 2 │ │ user 0,3,6... │ │ user 1,4,7... │ │ user 2,5,8... │ └────────────────┘ └────────────────┘ └────────────────┘
Estrategias de Sharding

🔢 Range-Based Sharding

Shard 0: user_id 1-1M, Shard 1: 1M-2M. Simple, pero puede crear hot spots (todos los nuevos usuarios van al mismo shard).

#️⃣ Hash-Based Sharding

shard = hash(user_id) % num_shards. Distribución uniforme. Problema: resharding es muy costoso si cambias el número de shards.

🗺️ Directory-Based Sharding

Un lookup service mantiene el mapa user_id → shard. Más flexible, puedes mover datos entre shards. Pero el lookup service es un SPOF.

🔄 Consistent Hashing

Los nodos y keys se mapean en un ring. Cuando agregas/quitas nodos, solo ~1/N de los datos se mueve. Solución al problema de resharding.

⚠️

Problemas del sharding: JOINs entre shards son costosos o imposibles. Transacciones cross-shard son complicadas. Resharding es una operación muy compleja. Solo shardea cuando realmente lo necesitas.

Replicación

Copiar datos entre nodos para alta disponibilidad y mejor rendimiento en reads.

LEADER-FOLLOWER (Master-Replica): [App]──Write──→[Leader DB]──sync──→[Follower 1] └──sync──→[Follower 2] [App]──Read───→[Follower 1 or 2] ✅ Simple de implementar ✅ Reads escalan horizontalmente ❌ Leader es SPOF para writes ❌ Replication lag (eventual consistency para reads) MULTI-LEADER: [App Europe]──→[Leader EU]──sync──→[Leader USA]──→[App USA] ✅ Writes en múltiples regiones ❌ Conflictos de escritura simultánea LEADERLESS (Cassandra, DynamoDB): Write va a N nodos simultáneamente. Quorum: W + R > N ✅ Máxima disponibilidad, sin SPOF ❌ Mayor complejidad, eventual consistency
Indexing

Estructuras de datos que aceleran las queries a costa de más storage y writes más lentos.

🌳 B-Tree Index

El más común. Excelente para range queries y equality queries. MySQL, PostgreSQL lo usan por default. O(log n) para reads y writes.

#️⃣ Hash Index

Solo para equality queries (WHERE id = 5). O(1) lookup. No sirve para range queries (WHERE id > 5). Usado en Redis internamente.

📊 Columnar Storage

Los datos se almacenan por columna, no por fila. Ideal para analytics (leer solo las columnas necesarias). Parquet, Redshift, BigQuery.

🔍 Full-Text Index

Inverted index: palabra → lista de documentos. Permite búsqueda por texto. Elasticsearch, PostgreSQL (tsvector), MySQL FULLTEXT.

-- Crear index compuesto (muy útil)
CREATE INDEX idx_user_created
ON posts(user_id, created_at DESC);

-- Esta query usará el index eficientemente
SELECT * FROM posts
WHERE user_id = 123
ORDER BY created_at DESC
LIMIT 20;

Connection Pooling

Uno de los optimizaciones más impactantes y menos discutida en entrevistas. Crítico para cualquier sistema a escala.

⚠️

El problema: Abrir una conexión a PostgreSQL cuesta ~50ms y consume ~10MB de RAM en el servidor. Si tenés 100 instancias de tu app con 10 threads cada una, son 1,000 conexiones simultáneas a la DB. Postgres empieza a degradarse severamente con más de 200-300 conexiones.

SIN CONNECTION POOLING: App Instance 1: [Thread 1 → DB conn] [Thread 2 → DB conn] [Thread 3 → DB conn] App Instance 2: [Thread 1 → DB conn] [Thread 2 → DB conn] [Thread 3 → DB conn] App Instance 3: [Thread 1 → DB conn] [Thread 2 → DB conn] [Thread 3 → DB conn] → 9 conexiones para 3 instancias de 3 threads → 1000 instancias = 10,000 conexiones ❌ CON PGBOUNCER (Connection Pooler): App Instance 1 ─┐ App Instance 2 ─┼──→ [PgBouncer] ──→ [PostgreSQL] App Instance 3 ─┘ (mantiene pool (solo 10-20 conex. ...1000 instancias de 20 conexiones siempre activas) (hasta 10,000 reutilizadas) conexiones lógicas) ✅ PostgreSQL ve solo 10-20 conexiones físicas ✅ Las 1000 instancias comparten ese pool ✅ Reducción de RAM en DB de 10GB → 200MB ✅ Latencia menor (sin handshake de nueva conexión)

PgBouncer

El pooler estándar para PostgreSQL. Tres modos: Session (1 conn por sesión), Transaction (1 conn por transacción — el más eficiente), Statement. Usar modo Transaction para APIs REST sin estado.

HikariCP (Java/JVM)

El connection pool más rápido para JVM. Default en Spring Boot. Pool de conexiones in-process. Configurar: maximumPoolSize, connectionTimeout, idleTimeout. Muy bajo overhead.

RDS Proxy (AWS)

Managed connection pooler para RDS/Aurora. Sin operación de PgBouncer. Integrado con IAM para auth. Reduce conexiones abiertas drásticamente en Lambda/serverless.

NewSQL — Lo mejor de los dos mundos

Bases de datos que combinan las garantías ACID de SQL con la escalabilidad horizontal de NoSQL.

El gap que llenan

SQL tradicional (PostgreSQL) escala verticalmente y es difícil de distribuir. NoSQL (Cassandra) escala horizontalmente pero sacrifica ACID. NewSQL da distribución geográfica, escalabilidad horizontal, y ACID completo.

Google Spanner

El first-mover. Usa relojes atómicos (TrueTime) para garantizar consistencia global. Escala a nivel planetario. Usado internamente en Google AdWords. Disponible como Cloud Spanner en GCP.

Open source. Inspirado en Spanner. PostgreSQL-compatible (mismos drivers, mismas queries). Multi-region nativo. Usado en Comcast, Bose, LaunchDarkly. Auto-sharding y rebalancing.

MySQL-compatible. HTAP: puede hacer OLTP y OLAP en la misma base. Open source. Muy popular en Asia (PingCAP). TiKV para storage distribuido, TiFlash para analytics columnar.

Database Migrations a Escala

Cómo hacer cambios al schema de producción con millones de filas sin downtime. Pregunta senior frecuente.

⚠️

El problema: ALTER TABLE users ADD COLUMN phone VARCHAR(20) NOT NULL DEFAULT '' en una tabla de 100M filas en PostgreSQL bloquea la tabla completa por minutos. Inaceptable en producción.

EXPAND-CONTRACT PATTERN (la solución estándar): OBJETIVO: Renombrar columna "email" → "email_address" PASO 1 — EXPAND (backward compatible): ALTER TABLE users ADD COLUMN email_address VARCHAR(255); ← Este ALTER es instantáneo (solo agrega columna NULL) App escribe en AMBAS columnas: email y email_address App lee de email (la vieja) — sin cambios al código de lectura PASO 2 — BACKFILL (en background, sin lock): UPDATE users SET email_address = email WHERE email_address IS NULL LIMIT 1000; ← En batches pequeños para no bloquear ← Correr en off-peak hours PASO 3 — MIGRATE reads: App ahora lee de email_address Sigue escribiendo en ambas PASO 4 — CONTRACT: App solo escribe en email_address ALTER TABLE users DROP COLUMN email; ← Ahora es seguro ───────────────────────────────────────────────────────── AGREGAR INDEX SIN LOCK (PostgreSQL): CREATE INDEX CONCURRENTLY idx_users_email ON users(email); ↑ "CONCURRENTLY" — construye el índice en background Puede tardar mucho más pero NO bloquea la tabla Reads y writes siguen funcionando durante la creación

Herramientas de Migration

Flyway/Liquibase: Versionado de migrations, rollback automático. Estándar en Java.
Alembic: Para SQLAlchemy (Python).
Sqitch: Agnóstico al lenguaje, git-like.
gh-ost (GitHub): Online Schema Change para MySQL. GitHub lo usa en tablas con miles de millones de filas.

Reglas de Oro

Siempre backward compatible — la vieja versión de la app debe seguir funcionando. Nunca renombres ni elimines en un solo step. Siempre usa CONCURRENTLY para nuevos índices en producción. Probar en staging con datos de volumen similar al de producción.
