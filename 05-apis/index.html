En una entrevista: Por default usa REST. Justifica GraphQL si el cliente necesita flexibilidad en qué datos pide. Justifica gRPC para comunicación interna entre microservicios donde la performance importa mucho.

REST — Mejores Prácticas

Diseño de URLs

Usa sustantivos (no verbos). Usa plural. Usa kebab-case. Versiona con /v1.

# ✅ Correcto
GET    /api/v1/users                    # Listar usuarios
GET    /api/v1/users/{id}               # Obtener usuario
POST   /api/v1/users                    # Crear usuario
PUT    /api/v1/users/{id}               # Reemplazar usuario completo
PATCH  /api/v1/users/{id}               # Actualizar parcialmente
DELETE /api/v1/users/{id}               # Eliminar
GET    /api/v1/users/{id}/orders        # Órdenes de un usuario

# ❌ Incorrecto
GET    /getUsers
POST   /createUser
GET    /api/users/getUserById?id=123

HTTP Status Codes

Usa los correctos — son parte del contrato de tu API.

# 2xx — Éxito
200 OK            # GET, PUT, PATCH exitosos
201 Created       # POST que crea un recurso
204 No Content    # DELETE exitoso

# 4xx — Error del cliente
400 Bad Request   # Input inválido
401 Unauthorized  # No autenticado
403 Forbidden     # Autenticado pero sin permiso
404 Not Found     # Recurso no existe
409 Conflict      # Conflicto (ej: email ya existe)
429 Too Many Reqs # Rate limit exceeded

# 5xx — Error del servidor
500 Internal Error # Error no esperado
503 Unavailable    # Servicio down (deploy, maintenance)
Rate Limiting

Proteger tu API de abuso y garantizar fair use.

HEADERS DE RATE LIMITING (estándar de la industria): Response Headers: X-RateLimit-Limit: 100 ← Máximo requests por ventana X-RateLimit-Remaining: 73 ← Requests restantes X-RateLimit-Reset: 1609459200 ← Timestamp cuando se resetea Cuando se excede el límite: HTTP 429 Too Many Requests Retry-After: 3600 ← Segundos para reintentar

Por IP

100 requests/hora por IP. Fácil de implementar pero fácil de evadir con proxies. Buen default para APIs públicas sin autenticación.

Por API Key / Usuario

Diferentes límites por plan: Free = 100/hr, Pro = 1000/hr, Enterprise = ilimitado. El modelo de monetización de la mayoría de APIs comerciales.

Autenticación y Autorización

🔑 API Keys

Simple. El cliente incluye una key en el header: Authorization: Bearer sk-abc123. Ideal para server-to-server. No expiran automáticamente (deben rotarse manualmente). OpenAI, Stripe, SendGrid usan este modelo.

🎫 JWT (JSON Web Tokens)

Token stateless con payload JSON firmado. El servidor no necesita almacenar sesiones. Contiene: header.payload.signature. Expira automáticamente (claim "exp"). Ideal para APIs de usuario. Riesgo: no se puede invalidar antes de expirar sin blocklist.

🔐 OAuth 2.0

El estándar para autorización delegada. "Login con Google/GitHub". El usuario autoriza a tu app a acceder a sus datos en otro servicio. Flujos: Authorization Code (web apps), Client Credentials (server-to-server), Device Code (smart TVs).

# JWT Structure
header = base64({
  "alg": "HS256",  # Algoritmo de firma
  "typ": "JWT"
})

payload = base64({
  "sub": "user_123",    # Subject (user ID)
  "iat": 1609459200,    # Issued at
  "exp": 1609545600,    # Expiration (24h)
  "roles": ["admin"]   # Claims adicionales
})

signature = HMAC_SHA256(header + "." + payload, secret_key)
WebSockets vs Server-Sent Events vs Polling

Para comunicación en tiempo real.

POLLING (el cliente pregunta repetidamente): Client: "¿hay novedades?" (cada 5 segundos) Server: "no" Client: "¿hay novedades?" Server: "no" Client: "¿hay novedades?" Server: "sí, aquí están" ❌ Muy ineficiente, alto overhead LONG POLLING: Client: "¿hay novedades?" → Server espera hasta tener algo (30 segundos después) Server: "sí, aquí están" → Client inmediatamente vuelve a preguntar ✅ Más eficiente que polling ❌ Sigue siendo un request por respuesta SERVER-SENT EVENTS (SSE): Client → Server: GET /events (una sola vez) Server → Client: data: {...} (cuando hay novedades) Server → Client: data: {...} ✅ Simple, unidireccional, reconnection automática ❌ Solo server → client, no bidireccional WEBSOCKETS: Client ←──── TCP connection ────→ Server Client → "mensaje" Server → "respuesta" Client → "otro mensaje" ✅ Bidireccional, baja latencia ❌ Más complejo, necesita manejo de conexión

Resource Modeling — Modelar bien tus recursos

Basado en HelloInterview · Los recursos deben representar cosas, no acciones.

💡

Regla de oro: Los recursos son sustantivos (events, tickets, users), no verbos (getEvent, createTicket). Las acciones las expresas con los métodos HTTP.

🔗 Path Parameters vs Query Parameters

Path parameters: Para identificadores obligatorios que definen el recurso.
/events/{id} → un evento específico
/events/{eventId}/tickets/{ticketId} → un ticket de un evento

Query parameters: Para filtros opcionales, ordenamiento y paginación.
/events?city=BuenosAires&date=2024-12-01&limit=20 

# ✅ API bien diseñada — estilo REST
GET    /api/v1/events                         # Listar eventos con filtros
GET    /api/v1/events/{id}                    # Un evento específico
POST   /api/v1/events                         # Crear evento
PATCH  /api/v1/events/{id}                    # Actualizar parcialmente
DELETE /api/v1/events/{id}                    # Eliminar
GET    /api/v1/events/{id}/tickets            # Tickets de un evento
POST   /api/v1/events/{id}/tickets/reserve    # Acción especial (verbo OK aquí)

# Request body para creación
POST /api/v1/events
{

Paginación — Offset vs Cursor

Cómo devolver listas grandes de datos de forma eficiente. Basado en HelloInterview.

🔢

Offset-based Pagination

Salta N registros y devuelve los siguientes M.

GET /posts?offset=20&limit=10

SELECT * FROM posts
ORDER BY created_at DESC
LIMIT 10 OFFSET 20

✅ Simple de implementar
✅ Permite saltar a cualquier página

❌ Inestable con datos en tiempo real (posts nuevos desplazan los resultados)
❌ OFFSET grande es lento en la DB

📍

Cursor-based Pagination ⭐

Usa un puntero opaco al último item. Más estable y eficiente.

# Primera página
GET /posts?limit=10
→ { data: [...], next_cursor: "eyJpZCI6MTIzfQ" }

# Siguiente página
GET /posts?limit=10&cursor="eyJpZCI6MTIzfQ"

SELECT * FROM posts
WHERE id < 123  -- cursor decoded
ORDER BY id DESC LIMIT 10

✅ Estable con datos en tiempo real
✅ Rendimiento constante (no OFFSET)
✅ Ideal para feeds infinitos

❌ No permite saltar a página arbitraria

🎯

En entrevistas: Si el sistema tiene datos que cambian frecuentemente (feeds, timelines), usa cursor-based. Para listados admin donde el usuario navega por páginas, offset está bien.

Autorización — RBAC (Role-Based Access Control)

Restringir endpoints según el rol del usuario. Patrón estándar en la industria.

RBAC — Roles y Permisos: ┌─────────────────────────────────────────────────────┐ │ User (autenticado) │ │ → GET /events ✅ Permitido │ │ → POST /events ❌ Denegado (403) │ │ → GET /events/{id} ✅ Permitido │ │ → DELETE /events/{id} ❌ Denegado (403) │ └─────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────┐ │ Manager │ │ → GET /events ✅ Permitido │ │ → POST /events ✅ Permitido │ │ → PATCH /events/{id} ✅ Solo sus propios eventos│ │ → DELETE /events/{id} ❌ Solo admin │ └─────────────────────────────────────────────────────┘ ┌─────────────────────────────────────────────────────┐ │ Admin │ │ → Todos los endpoints ✅ Acceso total │ └─────────────────────────────────────────────────────┘
# RBAC Middleware (FastAPI)
from functools import wraps

def require_role(*roles):
    def decorator(func):
        @wraps(func)
        async def wrapper(*args, current_user=Depends(get_current_user), **kwargs):
            if current_user.role not in roles:
                raise HTTPException(status_code=403, detail="Forbidden")
            return await func(*args, current_user=current_user, **kwargs)
        return wrapper
    return decorator

@app.post("/events")
@require_role("manager", "admin")
async def create_event(event: EventCreate, current_user=Depends(get_current_user)):
    return event_service.create(event, owner=current_user)
Idempotencia en APIs

Garantizar que el mismo request repetido produzca el mismo resultado. Crítico para pagos.

¿Qué métodos son idempotentes?

GET, PUT, DELETE: Idempotentes. Puedes repetirlos sin efectos secundarios.
POST: NO es idempotente. Cada llamada puede crear un nuevo recurso.
PATCH: Depende de la implementación.

Idempotency Key — Para operaciones críticas

El cliente genera un UUID único y lo envía en el header. Si el servidor ya procesó ese ID, devuelve la respuesta cacheada sin re-procesar. Stripe, PayPal y todos los procesadores de pago lo usan.

# El cliente genera un UUID único por intento
POST /api/v1/payments
Idempotency-Key: "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
{
  "amount": 9999,
  "currency": "ARS"
}

# Si el cliente reintenta (timeout, error de red), el servidor
# detecta que ese Idempotency-Key ya fue procesado y devuelve
# la misma respuesta sin cobrar dos veces
