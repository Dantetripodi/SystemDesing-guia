⚙️ Componentes Core

Los bloques de construcción que aparecen en casi todos los sistemas escalables.

Load Balancer

Distribuye el tráfico entrante entre múltiples servidores.

┌─────────────────┐ Clients ──────────→ │ Load Balancer │ (millions) └────────┬────────┘ │ ┌────────────────┼────────────────┐ ↓ ↓ ↓ [Server 1] [Server 2] [Server 3] (healthy) (healthy) (healthy) │ │ [Server 4] [Server N] (FAILED) ← LB detects failure, stops sending traffic

🔄 Round Robin

Cada request va al siguiente servidor en orden. Simple y efectivo cuando todos los servidores son equivalentes.

⚖️ Weighted Round Robin

Servidores más potentes reciben más tráfico. Configuras un peso por servidor (ej: 70/30).

📉 Least Connections

El request va al servidor con menos conexiones activas. Ideal para requests de larga duración.

🔗 IP Hash

El mismo cliente siempre va al mismo servidor. Útil para sesiones sin sticky sessions en el backend.

🏃 Least Response Time

Va al servidor más rápido actualmente. Requiere que el LB mida el response time continuamente.

🌐 Layer 7 (HTTP)

Puede rutear basándose en URL, headers, cookies. Ej: /api → API servers, /static → file servers.

Caching

Almacenar resultados frecuentes en memoria para reducir latencia y carga en la base de datos.

💡

Regla de oro: Si el 80% del tráfico accede al 20% de los datos, caching puede reducir la carga de tu DB en un 80%. Twitter sirve el 99% del timeline desde cache.

SIN CACHE: User → Server → Database (10-100ms) → Server → User (cada request pega en DB) CON CACHE: User → Server → Cache HIT (0.1ms) → User ← 99% de requests User → Server → Cache MISS → DB → Cache → User ← 1% Cache Hierarchy: CPU L1 Cache (0.5ns) → L2 (7ns) → RAM (100ns) → Redis (1ms) → DB (10-100ms)
⚡

Redis

In-memory data store. Soporta strings, hashes, lists, sets, sorted sets. Persistence opcional. Clustering. El más usado en producción.

Open Source · In-Memory
🐘

Memcached

Simple key-value cache. Más rápido que Redis para casos simples. Sin persistencia ni estructuras avanzadas. Multi-threaded.

Simple · Rápido
Estrategias de Cache

📥 Cache-Aside (Lazy Loading)

La aplicación maneja el cache. Read: busca en cache → si miss, busca en DB, guarda en cache. Write: invalida o actualiza cache. El más común.

# Cache-Aside pattern
def get_user(user_id):
    user = cache.get(f"user:{user_id}")
    if user is None:  # Cache miss
        user = db.query("SELECT * FROM users WHERE id = ?", user_id)
        cache.set(f"user:{user_id}", user, ttl=3600)
    return user

✍️ Write-Through

Cada write va PRIMERO al cache y LUEGO a la DB. Cache siempre consistente con la DB. Mayor latencia en writes, pero reads siempre desde cache.

💾 Write-Behind (Write-Back)

Writes van al cache inmediatamente, se propagan a DB de forma asíncrona. Muy rápido, pero riesgo de pérdida de datos si el cache falla antes de escribir en DB.

🔄 Read-Through

El cache maneja automáticamente el miss: busca en DB y se auto-popula. La app solo habla con el cache. Cache como abstracción completa.

Problemas de Cache

💥 Cache Stampede (Thundering Herd)

Muchas requests llegan al mismo tiempo con cache miss. Todas pegan en DB. Solución: mutex lock, probabilistic early expiration.

🌊 Cache Avalanche

Muchas keys expiran al mismo tiempo. Enorme carga súbita en DB. Solución: TTL jitter (aleatorizar expiración).

🔍 Cache Penetration

Requests para datos que no existen. Siempre miss en cache + miss en DB. Solución: cache el null result con TTL corto, Bloom filters.

Message Queues

Desacoplar productores de consumidores para procesar tareas de forma asíncrona.

SIN QUEUE (síncrono): Client → Server → [Enviar email] → [Resize imagen] → [Notificar push] → Response (espera 5 segundos) CON QUEUE (asíncrono): Client → Server → Queue.publish(task) → Response inmediato (100ms) ↓ [Worker 1: Envía email] [Worker 2: Resize imagen] [Worker 3: Notifica push] (se ejecutan en paralelo, el cliente no espera)

CDN (Content Delivery Network)

Red de servidores distribuidos globalmente que acercan el contenido al usuario.

SIN CDN: Usuario en Buenos Aires → Origin Server en Virginia → 150ms latencia CON CDN: Usuario en Buenos Aires → CDN Edge en São Paulo → 10ms latencia ↕ (si no está en cache) Origin Server en Virginia CDN POP (Points of Presence): ┌──────────────────────────────────────────────┐ │ 🌎 USA 🌍 Europa 🌏 Asia │ │ NY, LA London, Tokyo, │ │ Chicago Frankfurt Singapore │ │ │ │ → 100+ ubicaciones (Cloudflare: 285+) │ └──────────────────────────────────────────────┘

📤 Push CDN

Tú subes el contenido al CDN manualmente o automáticamente. CDN lo sirve desde ese momento. Mejor para contenido estático que cambia poco.

📥 Pull CDN

CDN busca el contenido en tu origen cuando alguien lo pide por primera vez. Luego lo cachea. Más simple de configurar. Primer request puede ser lento.

🌐

Proveedores principales: Cloudflare (285+ POPs), AWS CloudFront, Fastly, Akamai. Para imágenes y videos: Cloudinary, imgix. Netflix tiene su propio CDN: Open Connect.

API Gateway

Punto de entrada único que maneja cross-cutting concerns para todos los microservicios.

🔑

Autenticación centralizada

JWT validation, OAuth2, API Keys — una sola vez en el gateway, no en cada microservicio.

🛡️

Rate Limiting

Limitar requests por usuario/IP antes de que lleguen a los servicios. Protege contra DDoS y abuse.

Service Discovery

"¿Cómo se encuentran los microservicios entre sí?" — Pregunta que distingue el senior del semi-senior.

💡

El problema: En microservicios, las instancias se crean y destruyen continuamente (auto-scaling, deploys, fallos). Sus IPs cambian. ¿Cómo sabe el Order Service a qué IP llamar al Payment Service si cambia todo el tiempo?

CLIENT-SIDE DISCOVERY: Service Registry (Consul / etcd / Eureka): ┌──────────────────────────────────────────┐ │ payment-service → [10.0.1.5:8080, │ │ 10.0.1.6:8080, │ │ 10.0.1.7:8080] │ │ user-service → [10.0.2.1:8080] │ └──────────────────────────────────────────┘ Order Service: 1. "¿Dónde está payment-service?" 2. Consulta al Service Registry → lista de IPs 3. Elige una IP (load balancing en el cliente) 4. Hace el request directamente Cuando Payment Service arranca: → Se registra en el Registry (IP + health check) Cuando Payment Service muere: → Health check falla → Registry lo elimina ───────────────────────────────────────────────────────── SERVER-SIDE DISCOVERY: Order Service → API Gateway / Load Balancer → payment-service El cliente no sabe nada de service discovery. El gateway consulta al registry y hace el routing. ┌─────────────┐ "payment-service" ┌────────────────┐ │ Order Svc │──────────────────────→│ LB/Gateway │──→ 10.0.1.5 └─────────────┘ │ (Nginx/Envoy) │──→ 10.0.1.6 └────────────────┘──→ 10.0.1.7 ✅ El cliente es más simple ✅ El gateway puede hacer health checks, retries, circuit breaking

Consul (HashiCorp)

El más completo. Service registry + health checks + key-value store + service mesh. Multi-datacenter nativo. Usado en microservicios no-Kubernetes. Muy robusto en producción.

Kubernetes DNS

Si usás K8s, service discovery viene incluido. Cada Service tiene un DNS interno: payment-service.namespace.svc.cluster.local. El kube-proxy maneja el routing y load balancing automáticamente.

etcd

Distributed key-value store de alta consistencia. El "cerebro" de Kubernetes — almacena todo el estado del cluster. También usado directamente por servicios que necesitan un registro distribuido confiable.

Health Checks — El componente crítico

Liveness probe: ¿El proceso está vivo? Si falla → reiniciar el container. Ejemplo: GET /healthz → 200.
Readiness probe: ¿Está listo para recibir tráfico? Si falla → sacar del load balancer (pero no reiniciar). Ejemplo: verifica que la DB está conectada.
Startup probe: Para apps lentas en iniciar (JVM). Evita que Liveness mate el proceso antes de que levante.

Caching — Deep Dive

Basado en HelloInterview · Dónde cachear, qué evictar, y problemas en producción.

⚡

El impacto real: "Leer un perfil de usuario de Postgres toma ~50ms. Desde Redis en memoria toma ~1ms." Esa diferencia multiplicada por millones de requests es la diferencia entre escalar o no.

Dónde Cachear

🏎️ In-Process Cache

Cache dentro del proceso de la app. Acceso instantáneo (nanosegundos). Cada instancia tiene su propio cache — no compartido entre servidores. Ideal para: configuración, feature flags, datos muy pequeños y estables.

# Python in-process cache
from functools import lru_cache

@lru_cache(maxsize=1000)
def get_config(key: str) -> str:
    return db.get(key)

🌐 External Cache (Redis)

Cache compartido entre todos los servidores. ~1ms de latencia. Soporte para estructuras de datos complejas (lists, sets, sorted sets). Persistencia opcional. La solución más común para sistemas distribuidos.

📱 Client-Side Cache

Cache en el navegador o app móvil. HTTP caching headers (Cache-Control, ETag). Service Workers para PWA. Elimina requests completamente — zero latencia de red. Ideal para assets estáticos.

🌍 CDN Cache

Servidores distribuidos geográficamente. Reducen latencia de 250ms a 20ms para usuarios lejanos al origin. Modernamente pueden cachear respuestas de API, no solo archivos estáticos.

Políticas de Evicción

¿Qué pasa cuando el cache está lleno? El algoritmo determina qué se elimina.

🕐 LRU — Least Recently Used

Elimina el item que no se accedió hace más tiempo. Se adapta bien a la mayoría de workloads. El default en Redis. Bueno cuando los datos populares cambian con el tiempo.

📊 LFU — Least Frequently Used

Elimina el item menos accedido históricamente. Mejor que LRU cuando ciertos keys son siempre populares (resultados de búsquedas top, páginas de inicio). Mantiene el "long-term popular" content.

⏰ TTL — Time To Live

Cada key expira después de N segundos. Garantiza frescura de los datos. Casi siempre se combina con LRU/LFU. TTL corto = más consistencia, más carga en DB. TTL largo = mejor rendimiento, posible staleness.

📋 FIFO — First In First Out

Elimina el item más antiguo sin importar si fue accedido. Rara vez usado en producción. Útil solo cuando el orden de inserción importa más que la frecuencia de acceso.

🎲 Random Replacement

Elimina un item al azar. Sorprendentemente bueno en la práctica para algunos workloads uniformes. Sin overhead de tracking. CPU-efficient.

🔮 Allkeys-LFU (Redis)

En Redis: si la memoria está llena, aplica LFU a TODOS los keys (no solo los con TTL). Combinación práctica para maximizar cache hit rate en workloads variables.

Problemas Avanzados de Cache

🔥 Hot Keys

Un solo key recibe el 80% del tráfico (ej: el perfil de un celebrity). Sobrecarga un único nodo de Redis. Soluciones:
1. Key replication: user:beyonce:0, user:beyonce:1... elige uno al azar en reads.
2. Local fallback cache: cada servidor app cachea in-process el hot key por 5 segundos.
3. Rate limiting por key en el nivel de aplicación.

🔄 Cache Consistency — ¿Cuándo invalido?

Cuando un usuario actualiza su perfil, ¿cuándo actualizo el cache? Tres opciones:
Invalidar en el write: cache.delete("user:123") — simple, consistente, pero el próximo read pega en DB.
Actualizar en el write: cache.set("user:123", updated_data) — consistente, pero puede haber race conditions.
TTL corto: Dejar que expire solo en 60 segundos — eventual consistency aceptable.

# Ejemplo: Read-Through + Write-Invalidate (el más robusto)
def update_user(user_id, data):
    # 1. Actualiza en DB primero
    user = db.update(f"UPDATE users SET ... WHERE id={user_id}")

    # 2. Invalida el cache (no actualiza)
    cache.delete(f"user:{user_id}")

    # El próximo read va a poblar el cache fresco desde DB
    return user
