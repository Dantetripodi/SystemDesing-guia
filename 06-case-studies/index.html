🔬 Casos de Estudio

Diseña sistemas reales desde cero. Las preguntas más frecuentes en entrevistas de Google, Meta y Amazon.

1. URL Shortener (tipo bit.ly)

La pregunta más común en entrevistas. Testa: hashing, bases de datos, caching, escala.

📋 Requisitos

Funcionales: Dada una URL larga, generar URL corta. Dada una URL corta, redirigir a la original. Links expiran opcionalmente.
No funcionales: 100M URLs/día, reads/writes = 10:1, latencia < 10ms para redirect, 99.9% disponibilidad.

📐 Estimaciones

Writes: 100M / 86400 ≈ 1,160 QPS | Reads: 11,600 QPS | Storage: 500 bytes × 100M × 365 días × 5 años ≈ 900 GB

API Design: POST /api/v1/shorten → { longUrl, expireDate? } → { shortUrl } GET /{shortCode} → 302 Redirect a longUrl DELETE /api/v1/{shortCode} → Eliminar URL High-Level Design: ┌──────────┐ POST /shorten ┌──────────────┐ │ Client │──────────────────→│ Web Servers │ │ │←── shortUrl ──────│ (stateless) │ │ │ └──────┬────────┘ │ │ GET /{code} │ │ │──────────────────→ │ │ │←── 302 Redirect ── ↓ └──────────┘ ┌─────────────────────┐ │ Cache (Redis) │ │ shortCode → URL │ └──────────┬──────────┘ │ cache miss ↓ ┌─────────────────────┐ │ Database (MySQL) │ │ id | short | long │ └─────────────────────┘
Generación del Short Code

Opción 1: Hash + truncar (MD5/SHA256)

MD5(longUrl) → "1a79a4d60de6718e8e5b326e338ae533" → tomar primeros 7 chars → "1a79a4d". Problema: colisiones. Necesitas verificar si ya existe.

Opción 2: Base62 de un ID auto-increment ⭐ Mejor

Genera un ID único (1, 2, 3...) en la DB. Conviértelo a Base62 (a-z, A-Z, 0-9). ID 1 → "0000001", ID 100 → "0000028". 7 chars = 62^7 = 3.5 trillion URLs.

def to_base62(num: int) -> str:
    chars = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"
    result = []
    while num > 0:
        result.append(chars[num % 62])
        num //= 62
    return "".join(reversed(result)).zfill(7)

# ID 11157 → "2TX" (3 chars en base62)
print(to_base62(11157))  # "0002TX"

Deep Dive: Caching Strategy

Cache las 20% URLs más populares (80% del tráfico). TTL: 24 horas. Cache-Aside pattern. Si cache miss → DB → guardar en cache. Capacidad cache: 11,600 reads/s × 1KB × 0.2 = ~90GB de cache.

2. Twitter / Feed de Redes Sociales

La pregunta más compleja. Testa: feed generation, fanout, escala extrema.

📋 Requisitos

Funcionales: Publicar tweets, seguir usuarios, ver timeline (tweets de personas que sigues), likes/retweets.
No funcionales: 300M usuarios, 500M tweets/día, read-heavy (100:1), latencia timeline < 200ms.

FANOUT ON WRITE (Push model): Cuando alguien tweetea → push el tweet al feed de TODOS sus seguidores ┌──────────────┐ Tweet! ┌────────────────┐ │ User A │───────────→│ Tweet Service │ │ (10M seguidores) │ └────────┬────────┘ └──────────────┘ │ Fanout a 10M feeds (puede tardar minutos) ✅ Read rápido (feed pre-computado) ❌ Costoso para usuarios con muchos seguidores (celebrities) FANOUT ON READ (Pull model): Cuando alguien abre el app → pull tweets de todos los que sigue ✅ Simple, no hay fanout al publicar ❌ Lento (mucho trabajo en el read) SOLUCIÓN HÍBRIDA (Twitter lo usa): - Usuarios normales: Fanout on write - Celebrities (>1M seguidores): Fanout on read - Al leer el feed: mezcla = pre-computado + celebrities en tiempo real
DATA MODEL: ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ users │ │ tweets │ │ followers │ │ user_id PK │ │ tweet_id │ │ user_id │ │ username │ │ user_id FK │ │ follower_id│ │ created_at │ │ content │ │ created_at │ └──────────────┘ │ created_at │ └──────────────┘ └──────────────┘ ┌──────────────────────────────────┐ │ timeline_cache (Redis) │ │ Key: "timeline:{user_id}" │ │ Value: List of tweet_ids │ │ (últimos 800 tweet_ids) │ └──────────────────────────────────┘
3. Sistema de Chat (WhatsApp)

Testa: WebSockets, mensajería en tiempo real, presencia online, storage.

ARQUITECTURA CORE: [User A] ←── WebSocket ──→ [Chat Server A] [User B] ←── WebSocket ──→ [Chat Server B] │ ┌────────┴────────┐ │ Message Queue │ │ (Kafka) │ └────────┬────────┘ │ ┌────────┴────────┐ │ Message Storage │ │ (Cassandra) │ └─────────────────┘ FLUJO MENSAJE A → B: 1. A envía msg a Chat Server A (via WebSocket) 2. Chat Server A → Kafka → Chat Server B 3. Chat Server B entrega a B (si online) 4. Si B offline → push notification + mensaje en cola 5. B se conecta → recibe mensajes pendientes

Online Presence (¿Está conectado?)

Cada cliente envía un heartbeat cada 5 segundos. El servidor actualiza en Redis: "user:123:last_seen" = timestamp. Si el último heartbeat fue hace más de 30s → offline.

Message Storage

Cassandra es ideal: append-only, writes rápidos, sharding natural por conversation_id, TTL nativo para auto-expirar mensajes. Schema: (conversation_id, message_id, sender_id, content, timestamp).

4. Rate Limiter

Proteger APIs de abuso. Pregunta favorita de Amazon.

🪣 Token Bucket

Un bucket con capacidad N tokens. Se agrega 1 token cada X segundos. Cada request consume 1 token. Si no hay tokens, rechaza. Permite burst cortos.

⏱️ Sliding Window Counter

Cuenta requests en la última ventana de tiempo (ej: últimos 60 segundos). Más preciso que Fixed Window, que tiene burst en el límite de ventana.

# Rate limiter con Redis (Token Bucket)
import time
import redis

def is_allowed(user_id: str, limit: int = 100, window: int = 3600) -> bool:
    key = f"ratelimit:{user_id}"
    pipe = redis.pipeline()
    now = time.time()

    pipe.zremrangebyscore(key, 0, now - window)  # Limpiar entradas viejas
    pipe.zadd(key, {now: now})                  # Agregar timestamp actual
    pipe.zcard(key)                             # Contar requests en ventana
    pipe.expire(key, window)

    results = pipe.execute()
    return results[2] <= limit
5. Sistema de Video Streaming (YouTube)

Testa: CDN, video processing, storage a gran escala.

VIDEO UPLOAD FLOW: User → API Server → Raw Video Storage (S3) → Message Queue (Kafka) → Video Processing Workers: ● Transcoding (1080p, 720p, 480p, 360p) ● Thumbnail generation ● Metadata extraction → CDN distribution VIDEO STREAMING FLOW: User → API Server → CDN Edge Node (nearest) │ (cache miss) ↓ Origin Storage (S3) ADAPTIVE BITRATE STREAMING: Player detecta el ancho de banda del usuario: ● 4G rápido → 1080p ● 4G normal → 720p ● 3G → 480p ● Conexión lenta → 360p (cambia automáticamente durante la reproducción)
📊

Números reales de YouTube: 500 horas de video subidas por minuto. 1 hora de video = ~1.5 GB en raw. Después de transcoding a múltiples resoluciones = ~10 GB por video. CDN es CRÍTICO — sin él, la red no aguantaría.
