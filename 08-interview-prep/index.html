🎯 Preparación para Entrevistas

El framework, los errores comunes y las preguntas reales de Google, Meta y Amazon.

¿Qué Evalúa el Interviewer?

A diferencia del coding interview, system design es una conversación abierta.

✅ Lo que quieren ver

· Capacidad de clarificar requisitos antes de diseñar
· Razonamiento sobre trade-offs
· Cómo manejas la ambigüedad
· Conocimiento de componentes y cuándo usarlos
· Comunicación clara mientras diseñas
· Iterar y mejorar el diseño

❌ Lo que evitar

· Saltar a diseñar sin clarificar requisitos
· Una sola solución sin discutir alternativas
· No mencionar trade-offs
· Sobre-engineerear (YAGNI)
· Silencio — siempre piensa en voz alta
· Defender tu diseño cuando el interviewer sugiere cambios

El Framework de 7 Pasos

El proceso que usan los ingenieros de FAANG para estructurar su respuesta en 45 minutos.

TIMELINE DE 45 MINUTOS: [0-5 min] Paso 1: Clarificar requisitos [5-10 min] Paso 2: Estimaciones de escala [10-15 min] Paso 3: Definir APIs [15-20 min] Paso 4: Modelo de datos [20-30 min] Paso 5: Diseño de alto nivel [30-40 min] Paso 6: Deep dive en componentes [40-45 min] Paso 7: Bottlenecks y mejoras
1

Clarificar Requisitos (5 min)

NUNCA saltes este paso. Pregunta: ¿Cuántos usuarios? ¿Funcionalidades core vs nice-to-have? ¿Latencia aceptable? ¿Disponibilidad? ¿Es un MVP o sistema maduro? Repite los requisitos al interviewer para confirmar.

💬

Ejemplo: "Antes de comenzar, quiero clarificar algunos requisitos. ¿Estamos diseñando para ~100M usuarios? ¿El foco es en escritura o lectura? ¿Necesitamos soporte global o solo una región?"

2

Estimaciones de Escala (5 min)

Calcula QPS, storage, bandwidth. Los interviewers quieren ver que piensas en números, no vagamente en "grande". No necesitas ser exacto — órdenes de magnitud son suficientes.

# Ejemplo de estimación rápida
DAU = 100M usuarios activos diarios
# Writes:
write_QPS = 100M × 2 writes / 86400 seg ≈ 2300 QPS
# Reads (100:1 ratio):
read_QPS = 2300 × 100 = 230,000 QPS
# Storage (5 años):
storage = 2300 QPS × 500 bytes × 86400 × 365 × 5 ≈ 180 TB
3

Definir APIs (5 min)

Define los endpoints principales. Esto fuerza a ambos a estar de acuerdo en qué hace el sistema antes de hablar de implementación.

# Para un URL Shortener:
POST   /api/v1/shorten          # Crear short URL
GET    /{shortCode}             # Redirect
DELETE /api/v1/{shortCode}      # Eliminar
GET    /api/v1/{shortCode}/stats # Analytics
4

Modelo de Datos (5 min)

¿Qué entidades existen? ¿Relaciones? ¿SQL o NoSQL? Justifica tu elección. Menciona los índices principales.

-- Para un URL Shortener
CREATE TABLE urls (
  id         BIGINT PRIMARY KEY AUTO_INCREMENT,
  short_code VARCHAR(10) UNIQUE NOT NULL,
  long_url   TEXT NOT NULL,
  user_id    BIGINT,
  created_at TIMESTAMP,
  expires_at TIMESTAMP,
  click_count BIGINT DEFAULT 0
);
5

Diseño de Alto Nivel (10 min)

Dibuja el diagrama completo. Incluye: clientes, load balancers, servidores, cache, bases de datos, CDN, message queues. Explica el flujo principal de datos.

[Client] → [DNS] → [CDN] → [Load Balancer] ↓ [Web Servers x3] ↙ ↘ [Cache: Redis] [DB: MySQL] (Primary) ↓ [DB: MySQL] (Replicas)
6

Deep Dive en Componentes (10 min)

El interviewer pedirá profundizar en 1-2 componentes. Típicamente los más interesantes/complejos. Discute trade-offs, alternativas, y por qué elegiste esa solución.

💡

Prepara deep dives para: generación de IDs únicos, estrategia de caching, sharding de DB, manejo de failures, y consistencia de datos.

7

Bottlenecks y Mejoras (5 min)

¿Dónde falla primero el sistema? ¿Hay SPOF? ¿Cómo escalaría de 1M a 100M usuarios? ¿Qué sacrificaste y por qué? Demuestra visión crítica de tu propio diseño.

Errores Comunes y Cómo Evitarlos

❌ Error 1: No clarificar requisitos

El error más común. El interviewer da la pregunta deliberadamente vaga. "Design Twitter" — ¿cuáles features? ¿cuántos usuarios? Sin clarificar, puedes diseñar algo completamente distinto a lo esperado.

❌ Error 2: Silencio

Si estás pensando, di "Déjame pensar en esto un momento..." El interviewer necesita escuchar tu proceso de razonamiento, no solo la respuesta final.

❌ Error 3: Sobre-engeniería desde el día uno

No propongas Kafka, CQRS, y microservicios de entrada. Empieza simple. Luego itera cuando el interviewer pregunta "¿cómo escalarías esto?"

❌ Error 4: No discutir trade-offs

Cada decisión tiene pros y contras. "Elegí SQL porque garantiza ACID, pero el trade-off es que escalar horizontalmente es más difícil. Si necesitáramos más escala de writes, consideraría Cassandra."

✅ Lo que hace la diferencia

Interviewers reportan que los candidatos exitosos hacen esto: piensan en voz alta, identifican trade-offs espontáneamente, se anticipan a preguntas ("probablemente me preguntes sobre el sharding..."), y son receptivos a feedback.

Frases Útiles en la Entrevista

Expresiones que demuestran madurez de ingeniería.

Para Trade-offs

· "El trade-off aquí es X vs Y. Yo elegiría X porque..."
· "Hay varias opciones. Podríamos hacer A o B. A tiene más latencia pero B..."
· "Esto funciona para nuestro caso, pero si el volume fuera 10x, necesitaría..."

Para Mostrar Profundidad

· "Una cosa que me preocupa es el SPOF en este componente. Podríamos mitigarlo con..."
· "Si el cache se cae aquí, el sistema degrada gracefully porque..."
· "Para manejar el hot spot en usuarios con muchos seguidores, usaría..."

Semi-Senior vs Senior — ¿Qué busca el interviewer?

Las diferencias concretas que separan un 6/10 de un 9/10 en una entrevista de system design.

Dimensión Semi-Senior (pasa) Senior (destaca)
Requisitos Clarifica lo básico (usuarios, features) Pregunta por SLA, latencia aceptable, consistencia necesaria, presupuesto de infra, geografía de usuarios
Estimaciones Calcula QPS y storage Usa las estimaciones para justificar cada decisión ("con 10K QPS, un solo servidor no alcanza, necesito al menos...")
Trade-offs Menciona pros y contras cuando se lo preguntan Menciona trade-offs espontáneamente y ofrece alternativas: "Podría usar Kafka acá, pero si querés simplificidad SQS es suficiente porque..."
Failures Menciona que "hay que tener redundancia" Identifica el SPOF exacto, propone la solución específica (active-passive vs active-active), estima el impacto del failure
Escalabilidad "Agregaríamos más servidores" Identifica el bottleneck exacto (DB write throughput, network I/O, cache miss rate) y la solución específica para ese bottleneck
Seguridad No la menciona (o solo dice "usar HTTPS") Integra auth, autorización, rate limiting y encryption en el diseño desde el principio, no como afterthought
Datos Elige una DB y define el schema Justifica la DB con trade-offs, menciona indexing strategy, considera sharding si aplica, habla de migrations
Iteración Defiende su diseño inicial Evoluciona el diseño cuando el interviewer presiona: "Tenés razón, si la carga de reads es 10x, cambiaría X por Y porque..."
Señales de Staff Engineer Level

Para aspirar a roles Staff/Principal, el bar está más alto.

Preguntas que hacen los Staff Engineers

· "¿Qué pasa si esta región entera de AWS se cae?"
· "¿Cómo operamos este sistema en producción? ¿Qué paginamos?"
· "¿Cuál es el plan de migración si queremos cambiar la DB en 2 años?"
· "¿Cómo sabemos que el sistema está funcionando correctamente?" (observabilidad)
· "¿Cuánto costaría esto al mes?" (cost awareness)

Depth vs Breadth

Semi-Senior: Conoce muchos componentes superficialmente.
Senior: Domina los componentes core en profundidad + buenos trade-offs.
Staff: Todo lo anterior + puede diseñar sistemas completamente nuevos, identifica los problemas que el interviewer no mencionó, habla de impacto en el equipo y en la organización.
La diferencia no es saber más cosas — es entender más profundamente las que ya sabés.

Guía de Preparación por Nivel

Para nivel Semi-Senior (3-5 años exp)

Dominar los fundamentos: CAP, escalabilidad, latency numbers.
Componentes core sin dudar: Load Balancer, Redis, Kafka, CDN.
Casos de estudio: URL Shortener, Rate Limiter, Twitter Feed, Chat.
Framework de 7 pasos aplicado fluidamente.
Trade-offs básicos SQL vs NoSQL, REST vs GraphQL.

Para nivel Senior (5-8 años exp)

Todo lo anterior más profundidad en cada decisión.
Patrones avanzados: CQRS, Saga, Outbox, Strangler Fig.
Seguridad integrada: OAuth, mTLS, Zero Trust desde el diseño.
Operaciones: deployment strategies, RTO/RPO, chaos engineering.
Casos avanzados: Search engine, recomendaciones, geo systems.
Poder evolucionar el diseño bajo presión del interviewer.

Para nivel Staff/Principal (8+ años exp)

Todo lo anterior más impacto organizacional.
Hablar de cost optimization y tradeoffs de build vs buy.
Diseño multi-tenant, multi-region, compliance (GDPR, SOC2).
Mencionar how to operationalize: runbooks, on-call, SLOs/SLAs.
Considerar el impacto en otros equipos y en la organización.
Conocer cómo sistemas reales (Google, Meta, Netflix) resolvieron el problema.
