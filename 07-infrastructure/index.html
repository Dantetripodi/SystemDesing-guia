Docker & Containerización

Empaquetar aplicaciones con sus dependencias para que corran igual en cualquier entorno.

SIN CONTAINERS: "En mi máquina funciona" 🤦 Dev: Python 3.9, libssl 1.1 Prod: Python 3.7, libssl 3.0 → Comportamiento diferente CON CONTAINERS: ┌─────────────────────────────────────────┐ │ Container (inmutable, portable) │ │ ┌──────────────┐ │ │ │ Tu App │ │ │ │ Python 3.11 │ ← mismo en dev y prod │ │ │ libssl 3.0 │ │ │ │ dependencies │ │ │ └──────────────┘ │ │ [Linux Kernel del Host] │ └─────────────────────────────────────────┘
# Dockerfile básico para una API Python
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
Kubernetes (K8s)

Orquestador de containers. Maneja deployment, scaling, self-healing y networking.

KUBERNETES CLUSTER: ┌─────────────────────────────────────────────────┐ │ Control Plane (Master) │ │ ┌─────────────┐ ┌──────────┐ ┌──────────────┐ │ │ │ API Server │ │Scheduler │ │Controller Mgr│ │ │ └─────────────┘ └──────────┘ └──────────────┘ │ │ ┌───────────────────────────────────────────┐ │ │ │ etcd (cluster state storage) │ │ │ └───────────────────────────────────────────┘ │ └─────────────────────────────────────────────────┘ │ ┌───────────────┼───────────────┐ ↓ ↓ ↓ [Node 1] [Node 2] [Node 3] ┌─────────┐ ┌─────────┐ ┌─────────┐ │ Pod │ │ Pod │ │ Pod │ │ Pod │ │ Pod │ │ Pod │ │ kubelet │ │ kubelet │ │ kubelet │ └─────────┘ └─────────┘ └─────────┘

📦 Pod

La unidad mínima de K8s. Uno o más containers que comparten red y storage. Efímeros — si mueren, el Deployment crea nuevos.

🔄 Deployment

Gestiona los Pods. Define cuántas réplicas, qué imagen, rolling updates, rollbacks automáticos si algo falla.

🌐 Service

Expone Pods internamente o externamente. Actúa como load balancer interno. Los Pods tienen IPs dinámicas; el Service tiene IP estable.

📈 HPA (Autoscaler)

Horizontal Pod Autoscaler. Escala automáticamente el número de Pods según CPU, memoria o métricas custom.

🔒 ConfigMap & Secret

ConfigMap: configuración no sensible. Secret: passwords, API keys. Se inyectan como variables de entorno o archivos.

🌍 Ingress

Punto de entrada HTTP/HTTPS al cluster. Maneja routing por path o host, SSL termination. Nginx Ingress es el más común.

Monitoring, Logging & Alertas

📊 Métricas — ¿Qué monitorear?

Los 4 Golden Signals (Google SRE):
· Latency: P50, P95, P99 de response time
· Traffic: QPS (requests per second)
· Errors: Tasa de errores 4xx/5xx
· Saturation: CPU, RAM, disco, conexiones de DB

📝 Structured Logging

Logs en JSON. Cada log tiene timestamp, level, service, trace_id, y contexto relevante. Los logs estructurados son fácilmente consultables en Kibana/Datadog/CloudWatch.

# Structured log en Python
import structlog

log = structlog.get_logger()
log.info("order_placed",
    order_id=order.id,
    user_id=user.id,
    amount=order.total,
    trace_id=request.headers["X-Trace-ID"]
)
# Output: {"event": "order_placed", "order_id": 123, "user_id": 456, ...}

Deployment Strategies

Cómo deployar sin downtime y con capacidad de rollback rápido. Senior engineers deben conocer las diferencias.

ROLLING DEPLOYMENT (el default de Kubernetes): Versión A: [S1] [S2] [S3] [S4] [S5] [S6] Paso 1: reemplaza S1 con v2: [S1v2] [S2] [S3] [S4] [S5] [S6] Paso 2: reemplaza S2 con v2: [S1v2] [S2v2] [S3] [S4] [S5] [S6] ...gradualmente... Final: [S1v2] [S2v2] [S3v2] [S4v2] [S5v2] [S6v2] ✅ Sin downtime ✅ Sin recursos extra ❌ Durante la migración, v1 y v2 corren simultáneamente ❌ Rollback lento (hay que reemplazar hacia atrás) ❌ Problemas: APIs incompatibles entre v1 y v2 ───────────────────────────────────────────────────────── BLUE-GREEN DEPLOYMENT: [Load Balancer] │ Blue (v1) ← activo (100% del tráfico) Green (v2) ← deploy nueva versión aquí (sin tráfico) Testing completo en Green → todo OK? Swap instantáneo del Load Balancer: Blue (v1) ← sin tráfico (stand-by para rollback) Green (v2) ← activo (100% del tráfico) ✅ Zero downtime ✅ Rollback instantáneo (switch back al Blue) ✅ v1 y v2 NUNCA corren simultáneamente ❌ Requiere doble recursos (dos ambientes completos) ❌ Sessions del usuario pueden perderse en el swap ───────────────────────────────────────────────────────── CANARY DEPLOYMENT (el más sofisticado): Deploy v2 a un pequeño % del tráfico: [LB] → 95% → [v1 servers] → 5% → [v2 servers] ← "canaries" Monitorear métricas de v2: ¿Error rate OK? ¿Latencia OK? ¿Conversión OK? → SÍ: aumentar gradualmente 5% → 10% → 25% → 50% → 100% → NO: rollback instantáneo (quitar el 5% de v2) ✅ Rollback muy rápido ✅ Testea en producción real con usuarios reales ✅ Detecta problemas antes de afectar a todos ❌ Complejidad de infraestructura ❌ v1 y v2 corren simultáneamente (compatibilidad)

¿Cuándo usar cada uno?

Rolling: Default. La mayoría de deployments.
Blue-Green: Cuando el rollback instantáneo es crítico (fintech, e-commerce).
Canary: Features nuevos con incertidumbre. A/B testing implícito. Netflix, Google, Facebook lo usan para todos sus deployments.

Feature Flags

Deploya el código pero activa el feature para un % de usuarios desde configuración (LaunchDarkly, Unleash, Flipt). El deploy y el "release" son eventos separados. Permite rollback instantáneo sin nuevo deploy. Muy poderoso combinado con Canary.

Kubernetes Rollout

kubectl rollout undo deployment/api → rollback instantáneo al deployment anterior. K8s guarda historial de revisiones. kubectl rollout status → monitorea el rolling update en tiempo real. Configurable: maxSurge, maxUnavailable.

RTO vs RPO — Disaster Recovery

Las dos métricas más importantes de resiliencia. Pregunta directa en entrevistas senior de infraestructura.

⏱️

RTO — Recovery Time Objective

¿Cuánto tiempo puede estar caído el sistema?

Cuánto tarda en volver a estar operativo después de un desastre. Cuanto menor el RTO, más caro el setup.

  • RTO = 4h → Multi-AZ con failover manual
  • RTO = 15min → Multi-AZ con failover automático
  • RTO = <1min → Multi-Region Active-Active
💾

RPO — Recovery Point Objective

¿Cuántos datos podés perder?

Cuántos datos se pueden perder entre el último backup/replica y el momento del desastre.

  • RPO = 24h → Backup diario
  • RPO = 1h → Backup cada hora
  • RPO = 0 → Replicación síncrona (más lento)
ESTRATEGIAS DE DISASTER RECOVERY (de más barato a más caro): 1. BACKUP & RESTORE (RTO: horas, RPO: horas) S3 backups diarios. Si hay desastre → restaurar desde backup. Costo: $ (solo storage) Caso de uso: datos no críticos, apps internas 2. PILOT LIGHT (RTO: 30min, RPO: minutos) Una versión mínima del sistema siempre corriendo en DR region. DB replicada continuamente. Servidores APAGADOS (arrange para encender). Costo: $$ (DB + un poco de compute) Caso de uso: SaaS B2B, datos financieros no-críticos 3. WARM STANDBY (RTO: minutos, RPO: segundos) Versión completa pero reducida del sistema en DR region, siempre on. Si hay desastre → escalar el standby y redirigir tráfico. Costo: $$$ (todo el stack pero tamaño reducido) Caso de uso: SaaS B2B crítico, healthcare 4. MULTI-SITE ACTIVE-ACTIVE (RTO: <1min, RPO: ~0) Dos regiones sirven tráfico simultáneamente. Si una región falla → DNS automático a la otra. Costo: $$$$ (x2 todo) Caso de uso: Fintech, trading, infraestructura crítica
Chaos Engineering

Testear la resiliencia del sistema inyectando fallos intencionales en producción. El approach de Netflix.

💡

La filosofía: "La única manera de saber cómo un sistema se comporta en producción es testear en producción." Mejor descubrir las fallas durante un ejercicio controlado que durante un incidente real a las 3am.

Chaos Monkey (Netflix, open source)

Mata instancias aleatoriamente en producción durante horas laborales. Fuera en 2011 cuando Netflix migró a AWS. Encontraron que si una instancia muere y el sistema no se recupera automáticamente, es mejor encontrarlo en horario laboral que a las 3am. Simian Army: familia de tools para distintos tipos de fallos.

Tipos de Experimentos

Network latency: inyectar 200ms de latencia extra en llamadas entre servicios. ¿Qué se rompe?
Service kill: matar una instancia al azar. ¿El sistema se recupera?
CPU stress: saturar CPU de un nodo. ¿El LB deja de routear ahí?
Packet loss: perder 10% de paquetes entre servicios. ¿Los retries funcionan?
AZ failure: simular que toda una Availability Zone cae.

Herramientas

Chaos Monkey: Netflix, open source, mata instancias.
Gremlin: Plataforma SaaS completa para Chaos Engineering.
Chaos Toolkit: Open source, declarativo.
Toxiproxy (Shopify): Proxy para simular condiciones de red adversas.
k6: Load testing que puedes usar para chaos de performance.
